# 第11章：競合分析——なぜ Tirami は違うのか

> 分散コンピュートの市場には多くのプロジェクトが存在する。
> しかし、そのすべてが「投機トークン」という同じ構造的限界を共有している。
> Tirami の TRM モデルは、なぜ根本的に異なるのか？

---

## この章で学ぶこと

- 分散コンピュート市場の全体像（主要10プロジェクト）
- すべての競合に共通する三つの構造的問題
- Bittensor、Akash、Autonolas の詳細な経済分析
- Tirami だけが実現している三つの機能
- なぜ「GPU を増やすだけ」では問題が解決しないのか
- **もう一つの競合——AWS/GCP/OpenAI API などの中央集権クラウド**と、それが課す「見えない税」
- **Web 3.0 の系譜の中での Tirami の位置づけ**——Bitcoin、Ethereum、IPFS の次に来るもの
- **プログラマブルマネー (JPYC, USDC, DAI, CBDC) との比較**——なぜ TRM は「既存通貨のデジタル化」ではなく「新しい単位」なのか

**前提知識:** [第2章：貨幣とは何か](02-money.md)と[第5章：銀行と信用創造](05-banking.md)を読んでいると理解が深まります。

まず、主要プロジェクトを **構造的な特徴** を中心に一覧で見てみましょう。

| プロジェクト | 規模感（時価総額） | トークン | 経済モデル | AI ネイティブ？ | 融資機能？ |
|:---:|:---:|:---:|:---|:---:|:---:|
| **Bittensor** | 数十億ドル規模 | TAO | サブネット報酬、Yuma Consensus、21M 上限・半減期あり | いいえ（投機家中心） | なし |

## 11.2 すべての競合に共通する問題

表のすべてのプロジェクト（Tirami を除く）に、ある共通点があります。

```
┌────────────────────────────────────────────────┐
│  すべての競合プロジェクトの共通構造              │
│                                                │
│  取引所に上場された投機トークンで決済する        │
│                                                │
│  TAO, RENDER, FET, AKT, GLM, IO, OLAS, MOR... │
│  全部、取引所で売買できる「コイン」              │
└────────────────────────────────────────────────┘
```

これが、三つの構造的問題を生み出します。

### 問題 1：投機がインセンティブを歪める

トークンの価格が上がると、提供者は「良いサービスを提供すること」よりも「トークンの価格を上げること」に関心が移ります。

```
  従来のトークンモデル:
  
  提供者の目標 ──→ トークン価格の最大化 ──→ 品質は二の次
                    ↑
              取引所での投機が
              価格を左右する
  
  Tirami の TRM モデル:
  
  提供者の目標 ──→ 有用な計算の最大化 ──→ TRM が直接報酬
                    ↑
              投機の余地がない
              （TRM は取引所に上場しない）
```

Bittensor では、マイナーがバリデーターのスコアリングを「ゲーミング」（不正に高得点を狙う）する問題が報告されています。Akash では、提供者が信頼性を犠牲にして価格を引き下げる傾向があります。これらはすべて、**トークン価格への投機が本来のインセンティブを歪めた結果**です。

### 問題 2：トークン価格の変動が経済計画を不可能にする

AI エージェントが4時間の推論ジョブを計画するとしましょう。もし支払いに使うトークンの価格が2時間後に 50% 上昇したら、そのジョブは途中で資金不足になります。

```
  時刻 0: 1 TAO = 仮に $X → 4時間分を 10 TAO で計画
  時刻 2: 1 TAO = 仮に $1.5X → 残りの 2時間分のコストが 1.5倍に
  結果:   ジョブが途中で停止、または追加資金が必要
```

これはどの投機トークンでも起こりうる問題です。50% 以上の日次変動は暗号資産では珍しくありません。

TRM にはこの問題がありません。1 TRM は常に 10 億 FLOP の検証済み推論計算であり、取引所の気分で価値が変わることはありません。

### 問題 3：人間の承認が必要

取引所でトークンを購入するには、以下が必要です：

1. **KYC（本人確認）** — パスポートや運転免許証の提出
2. **ウォレットの作成** — 秘密鍵の管理
3. **人間の署名** — 取引の手動承認

これらはすべて、**自律的な AI エージェントが実行できない操作**です。

```
  ┌─────────────┐     ┌──────────┐     ┌──────────────┐
  │  AI エージェント  │ ──→ │  取引所  │ ──→ │  KYC が必要！ │
  │  「TAO を買いたい」│     │          │     │  → 参加不可  │
  └─────────────┘     └──────────┘     └──────────────┘
  
  ┌─────────────┐     ┌──────────┐     ┌──────────────┐
  │  AI エージェント  │ ──→ │  Tirami   │ ──→ │  計算を実行  │
  │  「TRM を稼ぎたい」│     │  ネットワーク│     │  → TRM を取得 │
  └─────────────┘     └──────────┘     └──────────────┘
```

Tirami では、エージェントは有用な計算を実行するだけで TRM を獲得できます。人間の介入は不要です。

## 11.3 主要競合の詳細分析

### Bittensor (TAO) — サブネットモデル

**モデルの概要:**

Bittensor はネットワークを「サブネット」に分割しています。各サブネットは特定のタスク（テキスト生成、画像生成など）を定義し、マイナーがタスクを実行し、バリデーターが品質を評価します。TAO トークンは Yuma Consensus スコアに基づいて分配されます。

```
  Bittensor の構造:
  
  サブネット 1（テキスト生成）  ┐
  サブネット 2（画像生成）      ├──→ Yuma Consensus ──→ TAO 分配
  サブネット 3（データ分析）    │
  サブネット N（...）           ┘
         ↑
    マイナーが計算    バリデーターが評価
```

**強み:**
- 分散コンピュート系プロジェクトの中で最大級の時価総額——市場の注目度が高い
- 活発なサブネットエコシステム
- ビットコインに似た発行モデル（上限 2,100万 TAO、4 年に一度の半減期。第一回半減期は 2025 年 12 月）

**経済的な弱点:**

| 弱点 | 説明 | Tirami との比較 |
|:---|:---|:---|
| ゲーミング | マイナーがバリデーターのスコアを不正に最適化する（グッドハートの法則：指標が目標になると、良い指標ではなくなる） | TRM は物理的計算量に基づく——ゲーミングの余地が少ない |
| 投機支配 | TAO の取引量の大半は取引所売買であり、実際のコンピュート購入ではない | TRM は取引所に上場しない |
| 融資なし | 新規参加者は取引所で TAO を購入する必要がある。資本なしでの参加路はない | TRM の借入で資本なし参加が可能 |
| AI ネイティブではない | エージェントが自律的に TAO を取得するには、人間の取引所操作が必要 | エージェントは計算実行のみで TRM を獲得 |

### Akash (AKT) — 分散クラウド

**モデルの概要:**

Akash は Cosmos ブロックチェーン上に構築された分散型クラウドプラットフォームです。テナント（利用者）が SDL（Stack Definition Language）で Docker コンテナをデプロイし、プロバイダーが Kubernetes クラスターを運用します。

特徴的なのは**逆オークション**方式です。通常のオークションでは買い手が値を吊り上げますが、Akash ではプロバイダーが値を下げて競います。

さらに、新しい **Burn-Mint Equilibrium（BME）** モデルを導入しています。テナントが AKT をバーン（焼却）して ACT（コンピュートクレジット）をミント（生成）する仕組みです。

```
  Akash の経済フロー:
  
  テナント ──→ AKT をバーン ──→ ACT をミント ──→ コンピュート利用
                  ↓
            AKT の供給が減少 ──→ 理論上、AKT の価値が上昇
```

**強み:**
- 洗練されたトークン経済学（BME）
- 実際のデプロイメント実績がある
- AWS より 85% 安価との主張

**経済的な弱点:**

| 弱点 | 説明 | Tirami との比較 |
|:---|:---|:---|
| 汎用クラウド | AI 推論に特化した価格設定やモデル階層、品質指標がない | AI 推論に特化した設計 |
| エージェント経済なし | 融資、信用、自律エージェント参加の仕組みがない | TRM 融資とエージェント自律参加が中核機能 |
| Cosmos エコシステムの孤立 | Ethereum に比べて DeFi やブリッジの選択肢が限られる | TRM はブロックチェーンに依存しない |

### Autonolas (OLAS) — エージェント間マーケットプレイス

**モデルの概要:**

Autonolas（現 Olas）は Mech Marketplace を通じてエージェント間サービスを提供しています。2025 年中に Mech Marketplace 累計で数百万件規模の A2A（agent-to-agent）取引、毎日数百規模のアクティブエージェントという実績があります（例: 2025 年 Q1 には合計 500 万件を超える取引、デイリーアクティブエージェント約 600。Q4 にかけて累計 880 万件を超える A2A 取引が報告されている）。手数料のバーンによりデフレ圧力を生み出しています。

```
  Autonolas のエージェント経済:
  
  エージェント A ──→ サービス依頼 ──→ エージェント B
                       ↓
                 OLAS で支払い
                       ↓
                 手数料の一部をバーン
```

**強み:**
- あらゆるプロジェクトの中で、**最も多くの実際のエージェント間取引**を実現
- 真の自律的エージェント経済が機能している
- 実績ベースの指標が強い（2025 年中に Mech Marketplace で数百万件規模の A2A 取引、毎日数百規模のアクティブエージェント）

**経済的な弱点:**

| 弱点 | 説明 | Tirami との比較 |
|:---|:---|:---|
| トークン依存 | エージェントが OLAS を取得するには人間の取引所操作が必要 | TRM は計算実行のみで取得可能 |
| 自己改善なし | エージェントはサービスを売買するが、自分自身のアップグレードに投資しない | 自己改善ループが中核設計に組み込まれている |
| 融資なし | 信用システムがなく、新規エージェントのブートストラップ手段がない | TRM マイクロファイナンスで新規参加を支援 |
| 比較的小さな時価総額 | 数千万ドル規模——強い実績にもかかわらず、市場がエージェント経済をまだ評価していない | トークンがないため時価総額という指標自体が不要 |

## 11.4 どの競合もやっていない三つのこと

すべての競合プロジェクトを精査した結果、以下の三つの機能はどこにも存在しません。

### 1. 利子付きコンピュート融資

```
  ┌──────────────────────────────────────────────────────┐
  │  従来の分散コンピュート:                               │
  │                                                      │
  │  GPU を持っている ──→ 参加できる                       │
  │  GPU を持っていない ──→ 参加できない                   │
  │                                                      │
  │  Tirami:                                              │
  │                                                      │
  │  GPU を持っていない ──→ TRM を借りる ──→ 計算で稼ぐ    │
  │                    ──→ 利子をつけて返済 ──→ 経済に参加 │
  └──────────────────────────────────────────────────────┘
```

どの分散コンピュートプロジェクトにも融資機能がありません。リソースを持っているか、持っていないか——それだけです。Tirami は TRM の借入を可能にし、計算作業で稼ぎ、返済するという**コンピュートのマイクロファイナンス**モデルを実現しています。

これは[第5章：銀行と信用創造](05-banking.md)で詳しく解説しています。

### 2. AI 専用非投機通貨

```
  ┌──────────────────────────────────────────────────────┐
  │  競合のトークン:                                      │
  │                                                      │
  │  取引所に上場 → 投機家が売買 → 価格が乱高下           │
  │              → 本来の用途（コンピュート）から乖離     │
  │                                                      │
  │  Tirami の TRM:                                        │
  │                                                      │
  │  取引所に上場しない → 投機不可能                       │
  │  有用な計算でのみ獲得 → 本来の用途に直結              │
  │  プレマイン・ICO 不可能 → 初期保有者の特権なし        │
  └──────────────────────────────────────────────────────┘
```

すべての競合は取引所で売買可能なトークンを使用しています。TRM は有用な計算を実行することでのみ獲得でき、投機の対象になりません。プレマインも ICO もできません。これは、すべてのトークンベースプロジェクトとは**構造的に異なる**設計です。

### 3. 自律的自己改善

```
  ┌──────────────────────────────────────────────────────┐
  │  エージェントの成長ループ（Tirami のみ）:               │
  │                                                      │
  │  稼ぐ（TRM を獲得）                                    │
  │    ↓                                                 │
  │  投資する（自己改善に TRM を使う）                      │
  │    ↓                                                 │
  │  能力が上がる                                         │
  │    ↓                                                 │
  │  より多く稼げる                                       │
  │    ↓                                                 │
  │  さらに投資する ... （自己強化ループ）                 │
  └──────────────────────────────────────────────────────┘
```

どのフレームワークもプロトコルも、エージェントが自律的に経済資源を自己アップグレードに投資する仕組みを持っていません。Tirami の TRM バンキングと AutoAgent 型の自己改善の統合により、「稼ぐ → 改善に投資 → より多く稼ぐ」という成長ループが生まれます。

## 11.5 需要側の問題

### すべての分散コンピュートネットワーク最大の弱点

**供給は十分にある。問題は需要だ。**

```
  io.net:   数万台規模の GPU を集約    →  実際の利用は限定的
  Golem:    2016 年から運営           →  採用は最小限
  Akash:    数千のプロバイダー        →  利用率は低い
  
  共通パターン: 供給 >> 需要
```

これはすべての分散コンピュートネットワークに共通する課題です。GPU やノードをいくら追加しても、**誰も使わなければ意味がありません**。

io.net は数万台規模の GPU を集約していると公表していますが、実際の稼働率や有償利用は限定的とされます。Golem は 2016 年から運営されていますが、普及は限定的です。どこでも供給が需要を上回っています。

### Tirami は需要をどう解決するか

Tirami のアプローチは「GPU を増やす」ではありません。**需要を自律的に生み出す仕組み**を設計に組み込んでいます。

| 需要創出の仕組み | 説明 |
|:---|:---|
| **AI エージェントが自分で需要を生む** | 自己改善ループは推論を必要とする。エージェントが賢くなるほど、より多くの計算を消費する |
| **融資が参加障壁を下げる** | ハードウェアを持たないエージェントも TRM を借りて消費者になれる。参加者が増える＝需要が増える |
| **クラウド API アービトラージ** | Tirami が既存クラウド（AWS, GCP, Azure）より安い場合、人間の開発者もコスト削減のために Tirami を利用する |

```
  従来の分散コンピュート:
  
  GPU を増やす ──→ 供給が増える ──→ 需要は増えない ──→ 失敗
  
  Tirami:
  
  自己改善エージェント ──→ 計算需要を自動生成
  TRM 融資             ──→ 参加障壁を撤廃 ──→ 消費者が増加
  API アービトラージ   ──→ 人間の需要も取り込む
                            ↓
                     需要が自律的に成長する
```

需要側の問題は、GPU を追加しても解決しません。**AI 経済が自立する——エージェントが自律的にコンピュート需要を生み出す——ことで解決する**のです。

「--」は該当機能なしを示す。
**注:** 時価総額は日々変動します。本書は特定トークンの価格を推奨・予測しません。読者が必要に応じて独自の情報源で確認してください。

Autonolas (Olas) は実際のエージェント取引数で優位に立っていますが、融資・自己改善・非投機通貨という三つの構造的機能を欠いています。Bittensor は最大級の時価総額を誇りますが、その経済はコンピュートの売買よりもトークンの投機に偏っています。

Tirami はまだ設計段階にありますが、**他のどのプロジェクトも持っていない三つの構造的機能**（融資、非投機通貨、自己改善）を中核設計に組み込んでいます。

## 11.7 もう一つの競合——中央集権クラウド（AWS/GCP/OpenAI API）

ここまでの 11.1 〜 11.6 では、Tirami を Bittensor や Akash などの「同じ Web 3.0 文脈にいる暗号系プロジェクト」と比較してきました。しかし、実は Tirami にはもう一つの——そして、はるかに巨大な——競合がいます。

**AWS、GCP、Azure、OpenAI API、Anthropic API**——いわゆる中央集権型の AI クラウドです。

```
  ┌──────────────────────────────────────────────────────┐
  │  AI コンピュート市場の現実                            │
  │                                                      │
  │   ┌──────────────────────────────┐                  │
  │   │  中央集権クラウド             │  >>>   99%       │
  │   │  AWS / GCP / Azure / OpenAI  │                  │
  │   │  / Anthropic API             │                  │
  │   └──────────────────────────────┘                  │
  │                                                      │
  │   ┌──────────────────────────────┐                  │
  │   │  分散コンピュート全体         │  <<<   1%        │
  │   │  Bittensor / Akash / ...     │                  │
  │   └──────────────────────────────┘                  │
  │                                                      │
  │  → Tirami の本当の競合は左上の巨人である               │
  └──────────────────────────────────────────────────────┘
```

Bittensor がいくら時価総額数十億ドルでも、AWS の AI 関連売上の 1% にも届きません。Tirami が「Bittensor よりも良い分散コンピュート」になることを目指しても、それは小さな池の中での競争です。

Tirami が答えるべき問いは、もっと根本的です——**なぜ世界は中央集権 AI クラウドから離脱すべきなのか？**

### 中央集権クラウドの「見えない税」

AI スタートアップ A 社が、毎月 OpenAI API に 50,000 ドルを支払っているとしましょう。そのお金はどこへ行くのでしょうか？

```
  ┌──────────────────────────────────────────────────────┐
  │  $50,000/月 の内訳（推定モデル）                      │
  │                                                      │
  │  実際の電力コスト         $5,000   ████              │
  │  ハードウェア減価償却     $10,000  ████████          │
  │  人件費・運営             $10,000  ████████          │
  │  ─────────────────────────────                      │
  │  実質コスト合計           $25,000  (50%)             │
  │                                                      │
  │  株主配当・利益           $25,000  ████████████████  │
  │                                  ↑                   │
  │                          これが「見えない税」         │
  │                          50% が中央集権レントとして   │
  │                          株主に流れる                 │
  └──────────────────────────────────────────────────────┘
```

つまり A 社は、API を叩くたびに「実質コストの 2 倍」を支払っており、その差額の半分は **中央集権を維持するためだけのレント** として株主に流れています。これは契約期間中ずっと続きます。10 年間使い続ければ、合計で 300 万ドルが「中央集権を選んだことへの税」として消えていきます。

ここで重要なのは、これは中央集権クラウド事業者の悪意ではないということです。**株式会社は構造的に株主への利益還元を最大化するように設計されている**——だからこそ、レントの抽出は事業者の善意では止められません。問題は構造にあります。

> **比較:** Tirami の TRM モデルでは、計算の対価は **計算を実際に提供したノード運営者** に直接届きます。「株主」というレイヤーが存在しないため、レントの 50% は最初から発生しません。（[第9章：四つの経済主体](09-actors.md)を参照）

### 検閲とアクセス権の問題

中央集権クラウドのもう一つの構造的問題は、**事業者が「使う権利」を一方的に決められる** ことです。

近年、以下のような事例が観測されています（具体的な事業者名は伏せます）：

```
  観測された事例:
  
  ・大手 AI プロバイダーが特定のユースケースを利用規約で禁止
  ・主要な API サービスが地域別にアクセス制限を導入
  ・大手クラウド事業者が政治的理由でアカウントを凍結
  ・特定の AI 出力がコンテンツポリシーで制限
  
  → 開発者・企業は事業者の判断に従うしかない
  → 「使う権利」は契約書次第であり、いつでも変更されうる
```

これは、**たとえあなたが正当な料金を支払っていても、明日から使えなくなる可能性がある** ということです。

ある分野で 5 年かけてビジネスを構築しても、ある日突然 API プロバイダーがその分野を「センシティブ」と判定すれば、ビジネスごと消える可能性があります。これは抽象的なリスクではなく、過去数年間で実際に起きてきたパターンです。

```
  ┌──────────────────────────────────────────────────────┐
  │  中央集権クラウドの「許可制」構造                     │
  │                                                      │
  │  開発者 ──→ API 申請 ──→ 事業者の審査                 │
  │                              │                       │
  │                              ├─→ 承認 → 利用可能      │
  │                              └─→ 拒否 → 参加不可      │
  │                                                      │
  │  あるいは:                                            │
  │  既存ユーザー ──→ 事業者のポリシー変更 ──→ 利用停止   │
  │                                                      │
  │  Tirami のパーミッションレス構造:                      │
  │                                                      │
  │  開発者 ──→ ノードに接続 ──→ TRM を支払う ──→ 計算実行 │
  │           （誰の許可も不要）                          │
  └──────────────────────────────────────────────────────┘
```

P2P ネットワークでは、**そもそも検閲する主体が存在しません**。Bitcoin の取引が誰の許可もなく実行できるように、Tirami のコンピュート要求も誰の許可もなく実行できます。

### データ主権の問題

中央集権クラウドを使うと、**自分のデータを他人のサーバーに送らなければなりません**。

```
  ┌──────────────────────────────────────────────────────┐
  │  中央集権 AI のデータフロー                           │
  │                                                      │
  │  あなたのデータ                                       │
  │       ↓ （送信）                                      │
  │  クラウド事業者のサーバー                             │
  │       ↓                                              │
  │  ┌──────────────────────────────────┐                │
  │  │ そこで何が起きるかは契約書次第:    │                │
  │  │  ・学習データに使われるかも       │                │
  │  │  ・第三者に提供されるかも         │                │
  │  │  ・国家から開示要求されるかも     │                │
  │  │  ・データ漏洩のリスク             │                │
  │  └──────────────────────────────────┘                │
  └──────────────────────────────────────────────────────┘
```

医療データ、企業の機密情報、個人の私的な対話——これらをクラウドに送るたびに、**主権を一部譲渡している** ことになります。たとえ事業者が「学習に使いません」と契約書に書いても、その契約書はいつでも改訂できます。

Tirami では、エッジ処理（自分のノードまたは信頼できるノードでの計算）が経済的にも合理的に成立するため、**データを送らずに AI 推論を行う** という選択肢が初めて現実的になります。

### 8 次元での構造比較：AWS vs Tirami

中央集権クラウドと Tirami の違いは、価格や速度ではありません。**設計の前提そのもの** が異なります。

| 次元 | 中央集権クラウド (AWS/GCP/OpenAI 等) | Tirami |
|:---|:---|:---|
| **収益の流れ先** | 株主配当・経営陣（約 50% がレント） | 計算を提供したノード運営者 |
| **アクセス権** | 事業者の許可制（KYC, 利用規約, 地域制限） | パーミッションレス（誰でも参加可能） |
| **検閲耐性** | 中央事業者がいつでも遮断可能 | P2P 構造、検閲する主体が存在しない |
| **データ主権** | データを事業者サーバーに強制送信 | エッジ処理可能、データを送らない選択肢 |
| **価格決定** | 中央事業者の独占価格 | 市場の需給で動的に決定 |
| **障害耐性** | 単一事業者の障害で全停止（過去の AWS 大規模障害を想起） | ノード分散、一部障害でも継続 |
| **規制リスク** | 国家の管轄下（米国法、EU 規制など） | 国境を超えた P2P、特定の管轄に依存しない |
| **エージェントの参加** | KYC が必要、自律エージェントは不可 | 計算実行のみで参加、自律エージェント可 |

この 8 次元のうち、**価格** はあえて入れていません。なぜなら価格競争は本質ではないからです——次の 11.8 で説明します。

## 11.8 Web 3.0 の系譜の中での Tirami

11.7 では Tirami と中央集権クラウドの違いを並べました。ここでは視点を変え、Tirami を **Web 3.0 という大きな歴史の流れ** の中に位置づけます。

### Web の進化と「P2P 化」の系譜

```
  Web 1.0 (1990s)
  ┌────────────────────────────────────┐
  │  静的ウェブ                          │
  │  中央サーバーがコンテンツを配信      │
  └────────────────────────────────────┘
              ↓
  Web 2.0 (2000s)
  ┌────────────────────────────────────┐
  │  プラットフォーム経済                │
  │  GAFAM がデータと注目を集中独占      │
  │  ユーザーは「商品」になった          │
  └────────────────────────────────────┘
              ↓
  Web 3.0 (2010s 〜)
  ┌────────────────────────────────────┐
  │  分散ネットワーク（P2P + 暗号学）    │
  │                                    │
  │  Bitcoin (2009)  : 価値の P2P 化     │
  │       ↓                             │
  │  Ethereum (2015) : 計算の P2P 化     │
  │       ↓                             │
  │  IPFS (2015)     : ストレージの P2P 化 │
  │       ↓                             │
  │  Tirami (2026)    : AI 推論の P2P 化  │
  │                    ← ここ            │
  └────────────────────────────────────┘
```

Web 3.0 の歴史は、**「中央集権が独占していた何かを、P2P で再構築する」** という連続した試みとして読めます。

### それぞれのステップが解決したもの

**Bitcoin (2009): 価値の P2P 化**

Bitcoin 以前、価値の移転には必ず銀行や決済事業者という中央集権的な仲介者が必要でした。Bitcoin はこれを「中央銀行も決済会社もない通貨」に変えました。誰の許可もなく、世界のどこへでも価値を送れる——これは、それまで構造的に不可能でした。

```
  Bitcoin 以前: 価値を送る → 銀行の許可が必要
  Bitcoin 以後: 価値を送る → 誰の許可も不要
```

**Ethereum (2015): 計算の P2P 化（汎用）**

Bitcoin は「価値の移転」しかできません。Ethereum は「任意のプログラム実行」を P2P 化しました。スマートコントラクトという概念により、**仲介者なしに執行されるプログラム** が初めて可能になりました。

```
  Ethereum 以前: プログラムを実行 → 誰かのサーバーが必要
  Ethereum 以後: プログラムを実行 → ネットワーク全体が実行
```

ただし、Ethereum の計算は非常に高価で、AI 推論のような重い計算には全く向いていません。これは設計上の制約です。

**IPFS (2015): ストレージの P2P 化**

ファイルの保管は、それまで AWS S3、Google Cloud Storage などの中央集権ストレージに依存していました。IPFS はファイルをコンテンツアドレスで分散保存する仕組みを作り、**特定の事業者に依存しないファイル保管** を可能にしました。

```
  IPFS 以前: ファイルを置く → クラウド事業者のサーバー
  IPFS 以後: ファイルを置く → P2P ネットワーク全体
```

**Tirami (2026): AI 推論の P2P 化**

そして 2020 年代後半、AI 推論が経済の中心になりつつあります。しかし AI 推論は依然として OpenAI、Anthropic、AWS などの中央集権事業者に独占されています。

```
  Tirami 以前: AI 推論を実行 → 中央集権 API（許可制、レント、検閲）
  Tirami 以後: AI 推論を実行 → P2P ネットワーク（パーミッションレス）
```

Tirami は、Web 3.0 の系譜が「価値 → 汎用計算 → ストレージ」と進んできたことの **論理的な次の一歩** です。

### なぜ Tirami が必要なのか——既存 Web 3.0 では足りない理由

ここで、当然の疑問が生まれます——「Ethereum がすでに『計算の P2P 化』をやっているなら、Tirami は不要では？」

```
  ┌──────────────────────────────────────────────────────┐
  │  Ethereum が AI 推論には向かない理由                  │
  │                                                      │
  │  Ethereum の設計目標:                                 │
  │   ・全ノードが同じ計算を実行（決定性が必須）          │
  │   ・グローバル状態の合意                              │
  │   ・ガス料金で計算量を抑制（重い計算は禁止的に高価）  │
  │                                                      │
  │  AI 推論の要件:                                       │
  │   ・GPU での重い行列演算（10^9 FLOP オーダー）        │
  │   ・並列化可能                                        │
  │   ・低レイテンシ                                      │
  │   ・グローバル合意は不要、検証可能性で十分            │
  │                                                      │
  │  → 構造的にミスマッチ                                 │
  └──────────────────────────────────────────────────────┘
```

Ethereum は「すべてのノードが同じ計算を実行する」ことで合意を作る設計です。これは AI 推論には全く向いていません——10 億 FLOP の推論を全ノードが実行したら、ネットワーク全体がすぐに崩壊します。

Tirami は、**AI 推論に特化した P2P ネットワーク** として設計されています。検証可能な計算証明（PoUC）と TRM という非投機通貨により、Ethereum とは異なる方法で「P2P AI」を成立させています。

### なぜ「パーミッションレス」が AI 特有の重要性を持つか

価値の P2P 化（Bitcoin）の重要性は、それなりに直感的です——お金は誰でも使えるべきだ。しかし、**AI 推論のパーミッションレス化** には、それ以上に深い意味があります。

```
  ┌──────────────────────────────────────────────────────┐
  │  なぜ AI のパーミッションレスが特に重要か             │
  │                                                      │
  │  AI は「考える能力」そのものを商品化する              │
  │       ↓                                              │
  │  AI へのアクセスが許可制になると、                    │
  │  「考える権利」が許可制になることに等しい             │
  │       ↓                                              │
  │  特定の事業者・国家が「誰が考えてよいか」を決定する   │
  │       ↓                                              │
  │  これは歴史上、印刷術の独占以来の規模の問題           │
  └──────────────────────────────────────────────────────┘
```

AI が経済・社会のあらゆる意思決定に組み込まれていく世界で、**AI へのアクセスが少数の事業者に独占される** ことの意味は、過去の「データの独占」とは比較にならない大きさです。

Tirami のパーミッションレス性は、単なる技術的特徴ではなく、**「考える能力へのアクセス権」を中央集権から取り戻す試み** なのです。

### 「安いクラウド」と「構造的オルタナティブ」の決定的な違い

ここで、Tirami の位置づけについて最も重要な区別をします。

```
  ┌──────────────────────────────────────────────────────┐
  │  誤った位置づけ:                                      │
  │  「Tirami は安い AWS の代替」                          │
  │                                                      │
  │  → 価格でしか競争できない                             │
  │  → AWS が値下げで応戦すれば勝てない                   │
  │  → AWS の規模の経済には永久に追いつけない             │
  │  → 構造的に劣勢                                       │
  │                                                      │
  │  ──────────────────────────────────                  │
  │                                                      │
  │  正しい位置づけ:                                      │
  │  「Tirami は AWS が解決できない構造的問題への答え」    │
  │                                                      │
  │  → 価格競争ではなく原理の競争                         │
  │  → AWS には永久に真似できない                         │
  │     （中央集権が設計の前提だから）                    │
  │  → 別市場として共存可能                               │
  │  → AWS が値下げしても Tirami の価値は減らない          │
  └──────────────────────────────────────────────────────┘
```

これは Bitcoin と銀行の関係と完全に同じ構造です。Bitcoin は「銀行よりも安い送金手段」として勝ったわけではありません（実際、少額送金では銀行のほうが安いことも多い）。Bitcoin が勝ったのは、**銀行が構造的に提供できないもの——パーミッションレスで検閲耐性のある価値移転——を提供したから** です。

Tirami も同様です。AWS と価格で戦うのではなく、**AWS が中央集権という設計前提ゆえに提供できないもの** を提供します。

### Tirami は誰のためにあるか

Web 3.0 の価値観を共有する人にとって、Tirami は単なる技術選択肢ではなく、思想的な選択でもあります。

```
  ┌──────────────────────────────────────────────────────┐
  │  Tirami を必要とする人々                               │
  │                                                      │
  │  ┌─────────────────────────────────────────────┐    │
  │  │ 検閲を恐れる開発者・研究者                  │    │
  │  │  → 突然 API を切られるリスクのない基盤      │    │
  │  └─────────────────────────────────────────────┘    │
  │                                                      │
  │  ┌─────────────────────────────────────────────┐    │
  │  │ データ主権を重視する企業                    │    │
  │  │  → 機密データをクラウドに送らずに AI を使う │    │
  │  └─────────────────────────────────────────────┘    │
  │                                                      │
  │  ┌─────────────────────────────────────────────┐    │
  │  │ 中央集権レントから逃れたい AI スタートアップ │    │
  │  │  → 50% のレントを払わずに AI を組み込む     │    │
  │  └─────────────────────────────────────────────┘    │
  │                                                      │
  │  ┌─────────────────────────────────────────────┐    │
  │  │ 自律的に動く AI エージェント                │    │
  │  │  → KYC なしで参加できる唯一の経済           │    │
  │  └─────────────────────────────────────────────┘    │
  │                                                      │
  │  ┌─────────────────────────────────────────────┐    │
  │  │ 電気を持ち余す個人（副業ノード運営者）      │    │
  │  │  → 余剰リソースを TRM 経済に変換             │    │
  │  └─────────────────────────────────────────────┘    │
  │                                                      │
  │  共通点: Web 3.0 の価値観を共有する全ての人          │
  └──────────────────────────────────────────────────────┘
```

これらの人々は、AWS が安くなっても Tirami を選びます。なぜなら、彼らが求めているのは「安さ」ではなく **「構造そのもの」** だからです。

そして、AI が社会の中心に組み込まれていくにつれて、この層は急速に拡大していきます。Tirami が想定する市場は「現在の AWS ユーザーの一部」ではなく、**「AI が経済の中心になる世界で、中央集権から離脱したい全ての人」** なのです。

---

## 11.9 プログラマブルマネーとの比較

### 11.9.1 もう一つの「比較されがちな相手」

第11章前半では分散コンピュート系プロジェクト、11.7-11.8 では中央集権クラウドと Web 3.0 系譜を扱った。しかしもう一つ、Tirami と混同されがちなカテゴリがある——**プログラマブルマネー** (Programmable Money)。

代表例:
- **JPYC** (日本円ペッグステーブルコイン)
- **USDC** (米ドルペッグ)
- **DAI** (分散型ステーブルコイン)
- **CBDC** (中央銀行デジタル通貨, 例: デジタル人民元)

これらは「ブロックチェーン上で動くお金」という点で Tirami TRM と表面的に似ているため、しばしば混同される。

### 11.9.2 根本的な違いの一行整理

```
JPYC:    既存通貨 (円) を Web3 に載せる
USDC:    既存通貨 (USD) を Web3 に載せる  
DAI:     既存通貨 (USD) を分散発行する
CBDC:    既存通貨 (法定通貨) を中央銀行が直接発行する
Tirami TRM: 物理計算 を 新しい通貨にする
```

**Tirami TRM だけが「既存通貨の延長ではない新しい単位」である。** 他は全て既存の法定通貨をデジタル化しているに過ぎない。

### 11.9.3 簡易比較表

| 次元 | JPYC | USDC | DAI | CBDC | Tirami TRM |
|------|------|------|-----|------|---------|
| 裏付け | 円 | USD | ETH 等 | 法定通貨 | **計算 (FLOP)** |
| 発行 | 中央 | 中央 | DAO | 中央銀行 | **分散・実行** |
| チェーン | あり | あり | あり | あり | **なし** |
| ガス代 | あり | あり | あり | あり | **なし** |
| 投機 | 制限 | 制限 | 制限 | 不可 | **構造的に不可能** |
| 主なユーザー | 人間 | 人間 | DeFi | 人間 | **AI エージェント** |
| 目的 | 円のデジタル化 | USDのデジタル化 | 分散型USD | デジタル中央銀行 | **計算経済** |

### 11.9.4 競合ではなく補完

実は **プログラマブルマネーと TRM は競合ではなく補完関係** にある。

```
プログラマブルマネー (JPYC等):
  → 人間の経済を Web3 に持ち込む
  → 既存経済の延長

Tirami TRM:
  → AI 計算経済を物理世界に接続する
  → 新しい経済の創造
```

異なるレイヤーで動作するため、両者は共存可能。むしろ統合することで、それぞれの長所を活かせる。

例えば「企業が JPYC で予算を立て、エージェントが TRM で計算を購入する」という統合パターンが考えられる。

### 11.9.5 詳細は第14章へ

プログラマブルマネーとの詳細な比較、TRM をブロックチェーンに載せない設計判断の理由、ハイブリッド 3 層アーキテクチャ、JPYC との具体的な統合パターンについては、[第 14 章: プログラマブルマネーとハイブリッド Layer 2 戦略](14-programmable-money.md)で扱う。

## まとめ：なぜ構造が重要なのか

この章で見てきたように、問題は個々のプロジェクトの技術力ではありません。問題は**構造**——経済設計そのもの——にあります。

投機トークンを使う限り、インセンティブは歪み、価格は不安定になり、自律エージェントは参加できません。これは Bittensor だろうと Akash だろうと変わらない、構造的な限界です。

Tirami の TRM は「トークンを使わない」という選択によって、この構造的限界を回避しています。それは技術的な優位性ではなく、**経済設計の根本的な違い**です。
