## 12.1 そもそも P2P とは何か

### 中央集権サーバーと P2P の違い

ほとんどのインターネットサービスは「クライアント・サーバー」モデルで動いています。
あなたのスマホ（クライアント）が、企業のデータセンター（サーバー）に接続する。
すべての通信、決済、データ保存はサーバーを経由します。

```
  ┌─────────────────────────────────────────────┐
  │           中央集権モデル                       │
  │                                              │
  │     [User1]   [User2]   [User3]   [User4]    │
  │        │        │         │         │       │
  │        └────────┼─────────┼─────────┘       │
  │                 │         │                  │
  │              ┌──▼─────────▼──┐               │
  │              │    SERVER     │  ← 全権力      │
  │              │ (企業が所有)   │  ← 全データ    │
  │              └───────────────┘  ← 全レント    │
  │                                              │
  └─────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────┐
  │              P2P モデル                       │
  │                                              │
  │     [Peer1]──────────[Peer2]                 │
  │       │  ╲          ╱   │                   │
  │       │   ╲        ╱    │                   │
  │       │    ╲      ╱     │                   │
  │       │     ╲    ╱      │                   │
  │     [Peer4]──╳──[Peer3]                     │
  │              ╳                              │
  │       ↑ 全員が対等                           │
  │       ↑ 中央が存在しない                     │
  │       ↑ レントを取る場所がない                │
  └─────────────────────────────────────────────┘
```

クライアント・サーバーモデルでは、中央サーバーを所有する企業が事実上すべての権力を持ちます。
価格を決め、ユーザーを排除し、データを売り、サービスを停止する権限はすべて中央事業者にあります。

P2P（Peer-to-Peer）モデルでは、参加者全員が対等な「ピア」として直接通信します。
中央が存在しないため、家賃（レント）を取る場所も、検閲する場所も、停止させる場所も構造的に存在しません。

これは単なる技術的選択ではなく、**経済構造そのものを規定する選択** です。

### P2P の歴史

P2P は突然現れた技術ではありません。約四半世紀の進化があります。

```
  1999 ── Napster ──────── 音楽ファイル共有（中央インデックス + P2P 転送）
            │
  2001 ── BitTorrent ────── 完全分散ファイル共有（DHT 発見、互恵的）
            │
  2009 ── Bitcoin ────────  P2P 価値交換（ブロックチェーン + PoW）
            │
  2015 ── IPFS ──────────── P2P コンテンツアドレス指定ストレージ
            │
  2020 ── libp2p / iroh ─── 汎用 P2P スタックの成熟
            │
  2026 ── Tirami ─────────── P2P AI 推論 + TRM 経済（本書のテーマ）
```

各段階で「P2P で何を交換するか」が変わってきました。

- Napster と BitTorrent は **ファイル** を交換した
- Bitcoin は **価値（マネー）** を交換した
- IPFS は **コンテンツ** を交換した
- Tirami は **計算（AI 推論）** を交換する

そして交換するものが変わるたびに、必要な経済モデルも変化してきました。
ファイル共有には経済が要らない（互恵性で十分）、価値の交換には PoW が必要、
そして「有用な計算」の交換には PoUW + TRM が必要——これが第10章で見た原理です。

## 12.2 なぜ Tirami は P2P でなければならないか

### 経済学的理由

第11章で見たとおり、Tirami の競合プロジェクトの多くは「分散型」を名乗りつつ、
実際には中央化された取引所・中央化されたガバナンスに依存しています。
Tirami が完全 P2P でなければならない経済学的な理由は三つあります。

**理由 1: 中央サーバーがあると、必ず誰かがそこからレント（家賃）を取る**

歴史上、中央化されたプラットフォームは例外なくレント抽出装置に変わります。
Uber は運転手から手数料を取り、Airbnb はホストから手数料を取り、
AWS は計算資源の真の提供者ではなく仲介者として価格を決めます。

中央サーバーが存在する限り、それを所有する誰かが「通行料」を取れる構造が生まれます。
Tirami が P2P でなければ、TRM 経済はやがて中央事業者のレント抽出装置に堕ちます。

**理由 2: 中央サーバーが価格を決めると、市場の力学が働かない**

中央事業者が価格を決定すると、それは「市場価格」ではなく「管理価格」になります。
管理価格は需給を正しく反映せず、第3章で見た価格メカニズムの自己組織化が機能しません。
Tirami の TRM 価格は、無数のピア間の P2P 取引が積み重なることで形成されます。
中央サーバーがあれば、この「分散的な価格発見」自体が成立しません。

**理由 3: 中央サーバーは検閲ポイントになる**

特定の AI モデルを禁止したい政府、特定のユーザーを排除したい企業、特定の取引を遮断したい規制当局——
中央サーバーがあれば、それらすべての要求は単一のポイントを押さえるだけで実行できます。
P2P では、検閲したい相手は「すべてのノード」を同時に押さえなければなりません。

### 技術的理由

経済学的理由とは独立に、純粋な技術的観点からも P2P は必要です。

**単一障害点（SPOF）の排除**

中央サーバーが落ちれば、サービスは止まります。
Tirami は「個人の遊休 GPU を世界規模で繋ぐ」ことを目指すため、
中央サーバーの停止が世界規模の AI 停止を意味する構造は許容できません。

**スケーラビリティ**

中央サーバー型のサービスは、ユーザー数が増えるたびに中央側の容量を増やす必要があります。
P2P では、参加者が増えれば資源も増える——これが「ネットワーク効果」と呼ばれる現象です。

**検閲耐性**

技術的にも、ノードが地理的・組織的に分散していれば、
特定の国や企業が単独でネットワークを停止することは事実上不可能になります。

## 12.3 P2P プロトコルの基礎

P2P プロトコルを構成する四つの基礎概念を順番に見ていきます。

### ノード発見（Discovery）

新しいピアがネットワークに参加するとき、最初の問題は「他のピアの存在をどうやって知るか」です。
中央サーバーがないので「サーバーに登録する」という解決策は使えません。

主な発見メカニズムは三つあります。

```
  1. Bootstrap nodes（既知の入り口）
     新参者は、最初に「既知のアドレス」リストに接続する。
     そこから他のピアの情報を貰う。

  2. DHT（Distributed Hash Table、分散ハッシュテーブル）
     ピア ID をハッシュ空間に配置し、責任範囲を分担する。
     Kademlia アルゴリズムが代表例。

  3. mDNS（multicast DNS）
     LAN 内でブロードキャストを使ってピアを発見する。
     インターネット越しには使えないが、家庭内・社内では極めて簡単。
```

Tirami の `tirami-net` は、現時点では **直接アドレス指定 + Welcome メッセージによる伝搬 + mDNS（LAN）** を組み合わせています。
将来的に DHT / WAN ディスカバリの実装が予定されています（v0.3 以降の課題）。

### トポロジー

ピアがどう接続されるかが「トポロジー」です。代表的な形は四つあります。

```
   メッシュ              リング              スター             DHT
                                                              （構造化）

   *───*               *───*                  *           *
   │╲ ╱│              ╱     ╲               ╱│╲          │
   │ X │             *       *             ╱ │ ╲         *──*
   │╱ ╲│              ╲     ╱             *  *  *       │  │
   *───*               *───*               中央           *──*
                                            ノード
   全員が全員と         隣だけと接続         一つを介す      ハッシュ空間
   接続する             情報は周回する        中央依存        で隣接定義
```

- **メッシュ**: 堅牢だが接続数が O(N²) で爆発する
- **リング**: 単純だがレイテンシが高い
- **スター**: 高速だが中央依存（P2P と矛盾）
- **DHT**: 構造化されたメッシュで O(log N) の計算量に抑える

Tirami は現状、**部分メッシュ + ゴシップ伝搬** に近いトポロジーを採用しています。
全員と接続するのではなく、一部のピアと接続し、情報はゴシップで広げる——という方式です。

### メッセージルーティング

ピア A からピア B へメッセージを届ける方法には二系統があります。

```
  直接接続（Tirami の基本方式）

   [A] ──────────────────────────► [B]
       QUIC 経由で直接トンネル
       NAT 越え (hole punching) は iroh が処理

  リレー / Onion routing（必要時）

   [A] ──► [Relay1] ──► [Relay2] ──► [B]
       中継ピアを経由
       匿名性 / NAT 不可越え対策
```

Tirami は基本的に **iroh QUIC による直接接続** を使います。
NAT 越えに失敗した場合、iroh が提供するリレーノードを介して接続を確立します。

### ゴシッププロトコル

ゴシップとは「ある情報を全ピアに広げるための噂話アルゴリズム」です。
中央サーバーがないため、新しい情報（取引記録、価格更新、ピアの加入）は「噂のように」広がります。

```
   t=0:   [A]知ってる
          [B][C][D][E][F] 知らない

   t=1:   [A]──►[B]
          [A]──►[C]
          → [B][C] も知った

   t=2:   [B]──►[D][E]
          [C]──►[F]
          → 全員が知った
```

ゴシップの特徴は二つあります。

1. **Eventual consistency（結果整合性）**: ある時点では全員の知識が一致しないが、時間が経てば収束する
2. **スケーラビリティの限界**: ノード数 N が大きくなると、ゴシップ自体の帯域消費が無視できなくなる

Tirami では、二者間で署名された取引（dual-signed trade）がゴシップで全網に伝搬されます。
これが Tirami 経済における「分散台帳」の実体です。
スケーラビリティ問題は Issue #5（混雑時の動的価格メカニズム）と密接に関係します。

`tirami-net/src/gossip.rs` にはレートリミット（毎秒 100 件）と重複排除（直近 100,000 件）が実装されており、
ゴシップ嵐（gossip storm）による DoS を防いでいます。

## 12.4 Tirami の P2P 実装

### iroh — 選ばれた P2P トランスポート

Tirami は P2P スタックとして [**iroh**](https://github.com/n0-computer/iroh) を採用しています。
iroh は「Number Zero（n0 computer）」が開発する P2P スタックで、QUIC をベースにした、
モダンで軽量な P2P トランスポートです。

なぜ libp2p ではなく iroh なのか？ 主な理由は四つあります。

```
  | 観点              | libp2p              | iroh                    |
  |------------------|---------------------|-------------------------|
  | 起源             | IPFS から派生         | n0 が一から設計          |
  | トランスポート    | TCP/QUIC/WebRTC等多数 | QUIC 中心                |
  | NAT 越え         | 外部ライブラリ依存    | 標準で hole punching     |
  | 暗号化           | TLS / Noise          | Noise（QUIC レイヤー）    |
  | コードサイズ      | 大きい（汎用）        | 小さい（焦点を絞った）     |
  | Rust エコシステム | 成熟                 | 急速に成熟中              |
```

iroh の選択理由を一言でまとめると **「Tirami に必要な機能だけを、シンプルに、高性能で提供してくれる」** からです。
libp2p は強力ですが汎用すぎて、Tirami には不要な機能と複雑性が多すぎました。

### QUIC + Noise の組み合わせ

iroh の通信レイヤーは二層構成です。

```
  ┌──────────────────────────────────────┐
  │  Tirami 独自プロトコル（FORGE_ALPN）     │  ← bincode メッセージ
  ├──────────────────────────────────────┤
  │  Noise Protocol Framework             │  ← 暗号化・認証
  ├──────────────────────────────────────┤
  │  QUIC（UDP 上の現代的トランスポート）  │  ← ストリーム多重化、低レイテンシ
  ├──────────────────────────────────────┤
  │  UDP                                   │
  └──────────────────────────────────────┘
```

- **QUIC**: TCP の後継。3-way handshake が不要で、接続確立が速い。1 接続上に複数ストリームを多重化できる。
- **Noise**: 暗号プロトコルの設計フレームワーク。WireGuard などでも採用されている、シンプルかつ強力な暗号化。

この組み合わせにより、すべての Tirami 通信は **暗号化された認証付きストリーム** として確立されます。
盗聴も改ざんもなりすましも、暗号学的に防がれます。

### ノード ID

Tirami のノード ID は **Ed25519 公開鍵そのもの** です。

```
  ノード起動時:
    1. Ed25519 鍵ペアを生成
    2. 公開鍵 = ノード ID
    3. 秘密鍵 = 自分のアイデンティティの証明
    4. 全ての送信メッセージは秘密鍵で署名
    5. 受信側は公開鍵（=ノード ID）で検証
```

これは Bitcoin のアドレスと同じ思想です。
「ID」は中央権威に登録するものではなく、暗号学的に自分自身が生成するものです。
誰かが他人になりすますには、その人の秘密鍵を盗む必要があります——
それは数学的に不可能な水準で困難です。

ノード ID が暗号鍵そのものであることの重要な帰結:

- 中央レジストリが不要
- なりすまし不可能
- 取引署名と通信認証が同一の鍵で行える
- ID = 鍵 = 経済主体——三つが一致する

`tirami-proto` の `Envelope` は必ず `sender: NodeId` を含み、QUIC 接続で認証された相手と一致しなければ拒否されます。
これが「すべての取引が dual-signed である」ことを技術的に支える基盤です。

### ピア発見

Tirami の現時点でのピア発見は次の段階を経ます:

```
  1. ユーザーが既知のピアアドレスを直接指定
       例: tirami worker --seed <public-key-hex>

  2. mDNS により LAN 内のピアを自動発見
       iroh の mDNS address lookup が処理
       (WAN 越えは iroh-canary の n0 DNS TXT record ベース)

  3. 接続後、相手から Welcome メッセージで known_peers を受信
       → そのリストから次の接続候補を得る（口コミ式の発見）

  4. Phase 19: PriceSignal.http_endpoint による HTTP エンドポイント広告
       → gossip で受信した PriceSignal に http_endpoint フィールドが
          付いていれば、その URL で OpenAI 互換 REST 経由でも
          そのピアに到達できる (select_provider → peer_http_endpoint
          で自動解決される)。iroh P2P と HTTP REST の両方の経路が
          開く。
```

将来的には DHT による広域発見、Bootstrap ノードの集合運用などが Issue として議論されています。

### HTTP → P2P フォワーディング (Phase 19 Tier C)

Phase 19 で Tirami は「HTTP API は local engine で model をロード済みでないと動かない」という制約を解消しました。ローカルエンジンにモデルがない状態で `POST /v1/chat/completions` を受けると、ノードは:

```
  1. peer registry から PriceSignal を持つ peer を探す
     (select_provider(model_id, max_tokens, local_node_id))
  2. その peer の http_endpoint または P2P iroh アドレスを解決
  3. PipelineCoordinator::request_inference で InferenceRequest を送信
  4. peer の handle_inference が推論を実行
  5. peer が TradeProposal を送信 (provider 署名つき)
  6. こちら側が counter-sign して TradeAccept 送信 → dual-signed trade
  7. TokenStream 受信 → HTTP response としてクライアントに返す
```

これにより **worker node (モデル未ロード) も OpenAI 互換 REST エンドポイントを提供でき、裏で自動的にメッシュの中から適切な provider を選んで forward する** という構造が完成しました。ユーザーアプリケーション側から見ると、ただの OpenAI API ですが、バックエンドは分散推論メッシュで dual-signed trade が走っています。

実装: `tirami-node/src/api.rs::forward_chat_to_peer` (Phase 19 Tier-C で追加)。Dispatcher ベースの TradeAccept ルーティングは `tirami-node/src/pipeline.rs::TradeAcceptDispatcher` (Phase 18.5-pt3 / #80 で修正)。

### 暗号化レイヤー

Tirami のすべての P2P 通信は Noise プロトコル（QUIC レイヤー）で暗号化されます。
これは「オプション」ではなく「強制」です。平文通信は存在しません。

なぜこれが経済学的に重要か:

- 取引内容（誰が誰に何を支払ったか）が第三者に見えない
- AI 推論のプロンプトと結果が盗聴されない（プライバシー）
- 中間者攻撃で偽の取引を注入できない（取引整合性）
- 検閲者が「Tirami トラフィック」をディープパケットインスペクションで識別しにくい

## 12.5 既存 P2P プロトコルとの比較

| プロトコル | 主な用途 | トランスポート | 発見 | 暗号化 | 経済モデル |
|---|---|---|---|---|---|
| **BitTorrent** | ファイル共有 | TCP | DHT (Kademlia) | なし（後付け） | なし（互恵性のみ） |
| **libp2p** | 汎用 P2P | TCP/QUIC/WebRTC 等 | DHT/mDNS | TLS/Noise | なし |
| **IPFS** | コンテンツ配信 | libp2p | DHT | TLS | Filecoin（別レイヤー） |
| **Bitcoin** | 価値の交換 | TCP | Seed nodes | なし | PoW |
| **Ethereum devp2p** | スマートコントラクト | TCP | discv5 (DHT) | RLPx | PoS |
| **Hypercore** | 同期データ構造 | UDP | DHT | あり | なし |
| **Tirami** | **AI 推論の交換** | **iroh QUIC + Noise** | **mDNS + 口コミ + 直接** | **Noise（強制）** | **PoUW + TRM** |

詳しい比較を見ていきます。

**BitTorrent** はファイルの「断片」を交換する目的に最適化されています。
経済モデルは「自分も配ればダウンロードも速くなる」という互恵性のみで、貨幣を必要としません。
これはファイル共有という用途には十分でしたが、AI 推論のような「一回きりの計算」には適用できません。

**libp2p / IPFS** は汎用 P2P 基盤として極めて強力ですが、それ自体には経済モデルがありません。
Filecoin は IPFS の上に経済を載せる試みですが、依然として投機トークンの域を出ていません（第11章参照）。

**Bitcoin** の P2P レイヤーは原始的（TCP、Seed nodes、平文）ですが、
経済モデル（PoW）が革命的でした。Tirami はその逆——
**経済モデル（TRM + PoUW）と P2P 技術（iroh）の両方を最新世代に置き換える試み** です。

**Ethereum devp2p** は discv5 という DHT ベースの発見と RLPx という暗号レイヤーを持ちます。
Tirami より発見プロトコルは成熟していますが、Tirami のように「個人の遊休 GPU の発見」には最適化されていません。

Tirami の独自性は二つあります。

1. **AI 推論専用**: 他のどの P2P プロトコルも、AI モデルの分散実行を前提に設計されていない
2. **TRM 経済が組み込み**: P2P プロトコル「と」経済モデルが分離していない

## 12.6 ネットワーク分断時の挙動

P2P ネットワークは構造的に「分断（partition）」が起こり得ます。
海底ケーブルの切断、国家ファイアウォール、ISP の障害——
これらが発生したとき、Tirami 経済はどう振る舞うのか？

### スプリットブレイン

ネットワークが二つの島に分断される状況を考えます。

```
  分断前:
    [A]──[B]──[C]──[D]──[E]──[F]──[G]

  分断発生（B-C 間のリンクが切れた）:
    [A]──[B]   ╳    [C]──[D]──[E]──[F]──[G]
        島1              島2

    島1 と島2 は相互に通信できない。
    どちらの島も「自分こそ Tirami ネットワーク」と認識し続ける。
```

### 局所的合意

このとき、各島の中では取引が普通に続行します。

- 島1 では `[A][B]` 間で TRM 取引が行われ、両者の台帳に署名付き取引が記録される
- 島2 でも `[C][D][E][F][G]` 間で取引が継続する
- ゴシップは島内でのみ伝搬する
- 価格は島ごとに独立に決まる——島1 の TRM 価格と島2 の TRM 価格は乖離し得る

これが **「結果整合性」と「局所的合意」** の本質です。
Tirami の経済は分断中も停止しません。各部分が独立に動き続けます。

### 復旧プロセス

分断が解消されたとき、何が起きるか？

```
  復旧瞬間:
    [A]──[B]──[C]──[D]──[E]──[F]──[G]
              ↑
              再接続

  起こること:
    1. 双方の Welcome メッセージで known_peers が交換される
    2. ゴシップが両島の取引履歴を相互に伝搬する
    3. 各ノードは知らなかった署名付き取引を順次受信
    4. 二者間で署名されている取引は、署名が有効なら全て受け入れる
    5. 価格は徐々に再収束する
```

ここで重要なのは、Tirami の取引は **二者間で完結する署名付き取引** であるという点です。
ブロックチェーンと違い、「全網のコンセンサス」を必要としません。
そのため分断中の取引も「無効化」されることがなく、復旧後にそのまま統合できます。

これは Bitcoin の reorg（分岐の巻き戻し）とは根本的に異なる設計です。
取引が二者間契約であるからこそ、分断耐性が極めて高い経済が成立します。

## 12.7 経済学との接続

P2P アーキテクチャは、Tirami 経済学の根幹に三つの形で接続しています。

### P2P トポロジーが価格形成に与える影響

物理的・ネットワーク的な近接性は、TRM 価格に直接影響します。

```
  例: ローカル LAN 内の取引

    [GPU 提供ノード] ───[1ms]──── [推論要求ノード]
                                   ↓
    レイテンシが極めて低い → 「価値が高い」 → 価格が上がる

  例: 大陸間の取引

    [日本のノード] ───────[150ms]──────── [欧州のノード]
                                            ↓
    レイテンシが高い → 「同じ計算でも価値が低い」 → 価格は下がる
```

これは第3章で見た価格メカニズムが、P2P トポロジーによって **空間的に変動する** ことを意味します。
中央サーバー型のクラウドでは隠蔽される地理的価格差が、Tirami では明示的に現れます。
ネットワーク密度（あるエリアにどれだけのピアが存在するか）が高い地域ほど、
価格収束が速く、価格差が小さくなる——これは Issue #5 で議論される動的価格メカニズムと連動します。

### ネットワーク効果と P2P スケーラビリティ

メトカーフの法則は「ネットワークの価値はノード数の二乗に比例する」と述べます。
P2P ネットワークではこれが特に強く効きます。

```
  ノード数:      10        100         1,000        10,000
  接続候補:     90        9,900       999,000      99,990,000
  価値比:        1         100        10,000       1,000,000
```

ただし、純粋なメッシュトポロジーは O(N²) の接続数になり、現実的に維持できません。
Tirami のように **部分接続 + ゴシップ伝搬** とすることで、各ノードの負荷を O(log N) ～ O(√N) に抑えつつ、
経済的な接続性は維持できます。

これが意味すること: **Tirami は参加者が増えるほど、各ノードのコスト負担が劇的に増えるわけではない**。
ネットワーク効果のメリットだけを享受しつつ、スケーラビリティの罠を避ける設計です。

### エッジコンピューティングと P2P の親和性

「エッジコンピューティング」とは、データセンターではなくユーザーの近くで計算を行う発想です。
P2P は、この発想と本質的に親和的です。

```
  クラウド型:
    [ユーザー] ──[100ms]──► [リモートデータセンター] ──► 計算
    遠い、遅い、高コスト、データセンター事業者にレント

  Tirami / P2P + エッジ:
    [ユーザー] ──[2ms]──► [近所の遊休 GPU ノード] ──► 計算
    近い、速い、低コスト、GPU 提供者に直接支払い
```

個人や中小事業者の遊休 GPU が「ネットワーク的に近い」位置にあるとき、
それは大手データセンターに対して構造的な優位性を持ちます。
P2P 発見メカニズム（mDNS、近接ピア発見）はこの優位性を技術的に実現するための前提です。

これは Issue #7（個人ノードの参入経路）と直接つながります。
個人ノードの経済的な存在意義は、エッジでの低レイテンシ提供——
そしてこれを成立させるのが P2P アーキテクチャです。

## 12.8 残された課題

Tirami の P2P レイヤーは、まだ完成していません。v0.3 以降に向けて以下の課題があります。

- **スケーラビリティの限界**: 現在のゴシップ実装は毎秒 100 件のレートリミットを持つ。N が極めて大きくなったときの伝搬遅延と帯域消費は未解決
- **広域ノード発見の効率化**: mDNS は LAN のみ。WAN での DHT ベース発見は未実装
- **ネットワーク分断からの復旧時間**: 分断が長期化したときのゴシップ再同期のコストは未測定
- **ピア評判システム**: 悪意あるピアを経済的に排除するメカニズム（Sybil 耐性）は今後の課題
- **リレーノードの分散化**: 現状 NAT 越えに iroh のリレーを利用する。これ自体の分散化が課題

これらは Tirami 経済学の発展と並行して、`tirami-net` の進化として取り組まれていきます。

---

## まとめ

P2P アーキテクチャは Tirami 経済学の **前提条件** です。装飾でも代替可能な技術選択でもありません。

- 中央サーバーがあれば、必ずレントが取られる → Tirami の「レントゼロ」の前提が崩れる
- 中央サーバーがあれば、価格は管理価格になる → Tirami の「分散的な価格発見」が機能しない
- 中央サーバーがあれば、検閲ポイントが存在する → Tirami の「検閲耐性」が消失する

逆に言えば、P2P だけでも経済は成立しません。
BitTorrent には経済が要らず、libp2p には経済がない——
P2P と TRM 経済が組み合わさって初めて、「分散型 AI 経済圏」が成立します。

```
   P2P 技術               TRM 経済モデル
   （tirami-net）           （tirami-economics）
       │                       │
       └───────────┬───────────┘
                   │
              不可分な統合
                   │
            ┌──────▼──────┐
            │    Tirami    │
            │ 分散 AI 経済 │
            └─────────────┘
```

Tirami は技術プロジェクトでも経済プロジェクトでもなく、両者の **接続点** にあります。
そしてこの接続点こそが、本書のすべての章が指し示してきた場所です。
