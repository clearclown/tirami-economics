## なぜ「残された課題」を論じるのか

新しい経済システムを提案するとき、よくある誤りは「自分の設計は完璧である」と主張することです。
Tirami はそれを避けます。

理由は二つあります。

**一つ目: 誠実さ。**
不完全な設計を完璧であるかのように語ることは、読者を欺くことになります。
学問の世界でも、ビジネスの世界でも、自らの限界を語れない理論は信用されません。

**二つ目: 進化可能性。**
弱点を明示することで、コミュニティが改善を提案できます。
これは Tirami が掲げる「permissionless」「オープンソース」という思想の自然な延長です。
中央集権型のシステムは、自社の弱点を隠す動機を持ちます。
P2P 型のシステムは、弱点を共有することで強くなります。

```
  中央集権型システム:
    弱点を隠す → 競合に対する優位性を保つ
    → 結果: 弱点が放置され、ある日突然崩壊する
  
  P2P 型システム:
    弱点を共有する → コミュニティが改善する
    → 結果: 弱点が継続的に修正され、進化する
```

この章では、Tirami に対して提起された 5 つの主要な批判を取り上げ、それぞれに正直に答えます。
批判の出典は、すべて [`tirami-economics` GitHub リポジトリの Issues #4〜#8](https://github.com/clearclown/tirami-economics/issues) です。

各節は次の構造で書かれています:

1. **批判**: 何が問題とされているか
2. **Tirami の現状**: 現時点での設計と防御
3. **P2P / Web 3.0 視点からの応答**: なぜそれが「弱点」ではないか、あるいは部分的に防御できるか
4. **残る課題**: それでも未解決の問題と、今後の研究方針

## 13.1 馴れ合い攻撃（Collusion）への耐性

### 批判（Issue #4）

Tirami の TRM 発行は「双方署名」によって正当性が担保されています。
依頼者と実行者の双方が署名することで、「実際に計算が行われた」という事実が暗号学的に裏打ちされる、という設計です。

しかし、ここに穴があります。

```
攻撃シナリオ:
  ノード A と ノード B が結託する
    ↓
  実際には計算をしていないのに、
  「A が依頼し、B が実行した」という双方署名 TRM を生成する
    ↓
  この偽 TRM をネットワークに流す
    ↓
  偽 TRM が実需 TRM と区別できなければ、貨幣供給が水増しされる
```

これは「シビル攻撃 + 馴れ合い攻撃」のハイブリッドであり、PoW のように物理的計算を伴わない暗号通貨では、典型的な脆弱性パターンです。
Tirami の双方署名は「物理的計算が裏付けとして存在する」という前提に立っていますが、その前提自体を結託で破ることができます。

### Tirami の現状

Tirami は完全な防御を持っていません。
現時点での部分的な防御は次の三つです:

1. **物理的コストの裏付け**: 実際に計算をすれば電力コストが発生する。偽 TRM は電力コストなしで作れるため、コスト構造が異常になる
2. **レピュテーション**: ノードペアの過去の取引履歴
3. **抜き打ち再計算**: 一部の TRM について、第三者ノードが再計算を行って整合性を確認する仕組みの草案

しかし、これらは「攻撃を完全に防ぐ」ものではなく、「攻撃のコストを上げる」ものです。

### P2P 経済の自己防衛メカニズム

Issue #4 のコメントで提示された再考の核心は次の点です:

> 「偽 TRM を作ること」と「その TRM を使うこと」は別問題。
> P2P 市場では後者で攻撃が無効化される可能性がある。

つまり、攻撃者が偽 TRM を生成できたとしても、**他のノードがその TRM を受け取らなければ、攻撃は経済的に無意味になる**ということです。

```
  攻撃者 A と B が結託 → TRM を相互生成
                ↓
  他のノードが A と B の取引パターンを観測
                ↓
    ・異常な取引頻度（A↔B のみ）
    ・他のノードとの取引が皆無
    ・レピュテーションの偏り
                ↓
  他のノードが A・B との取引を拒否
                ↓
  A・B の TRM は「市場で受け入れられない通貨」になる
                ↓
  攻撃のインセンティブが消失
```

これは Bitcoin の 51% 攻撃と同じ構造です。
Bitcoin でも理論的には 51% の計算力で偽トランザクションを作れますが、市場（取引所、ウォレット、エコシステム）が受け入れなければ、その「偽 BTC」には何の価値もありません。
攻撃者は莫大な計算力を費やして、無価値な通貨を作るだけです。

### Issue #4 で議論された具体的対策

Issue #4 のコメントでは、次のメカニズムが提案されています:

- **取引パターンの異常検知**: ノードペア間の取引集中度を監視する。A↔B のみで取引が完結している場合は、結託のシグナルとして扱う
- **ソーシャルグラフ分析**: 結託グループは小さなクリーク（密に結ばれた小集団）を形成しがち。グラフ理論によって構造的に検出可能
- **市場拒否メカニズム**: 信頼スコアの低いノードからの TRM を、選択的に受け取らない
- **暗号学的検証 + 経済的防衛のハイブリッド**: 完全な暗号証明（zkML 等）はコストが高いので、経済的圧力で補完する

これは「中央権威による検証」ではなく「分散的な合意による検証」という、Web 3.0 の根本思想と一致します。

### 残る課題

```
[ ] 抜き打ち品質検証メカニズムの正式設計
[ ] zkML（ゼロ知識機械学習証明）の実用性検証
[ ] TEE（Trusted Execution Environment）の活用検討
[ ] ソーシャルグラフ異常検知アルゴリズムの実装
[ ] 結託コストと検出確率の定量モデル
[ ] 実証実験（数百ノード規模での攻撃シミュレーション）
```

**緊急度: 高。** これは Tirami の貨幣としての健全性そのものに関わる問題です。
v0.3 では、最低限「経済的圧力による拒否メカニズム」のプロトコルレベル実装を目指します。

## 13.2 ゴシッププロトコルのスケーラビリティと価格収束の遅延

### 批判（Issue #5）

Tirami は中央オーダーブックを持ちません。
価格情報はゴシッププロトコル（噂話のように、ノード同士が情報を交換する仕組み）によって伝播します。
このアーキテクチャには二つの潜在的な問題があります。

1. **スケーラビリティ**: ノード数が数百万規模になると、ゴシップトラフィックが爆発する
2. **価格収束の遅延**: ローカルな価格が異なるため、ある瞬間にネットワーク全体で「同じ価格」が成立しない

特に二点目は、金融市場の常識からすると深刻な問題に見えます。
NASDAQ は全参加者が同じオーダーブックを見るからこそ、価格発見が機能する。
ローカルに価格が分裂していれば、「裁定機会」が永久に存在し、効率的市場仮説が破綻するのではないか?

### Tirami の現状

- ゴシッププロトコル（gossipsub 系）による eventual consistency
- 数千ノード規模では実装上の問題はない（[`tirami-net` クレート](https://github.com/clearclown/tirami)で検証済み）
- ローカル価格の分散は、設計上「許容される」

### ローカル価格は「バグ」ではなく「機能」

Issue #5 のコメントで提示された再考の核心はこうです:

> グローバル価格収束を理想視するのは中央集権的な発想。
> P2P ネットワークでは「ローカル市場」の存在そのものが特徴であり強みである。

```
  中央集権（NASDAQ 等）:
    全参加者が同じオーダーブックを見る
    → 1 つのグローバル価格
    → 物理的距離が HFT の優位性に変換される
    → 取引所に物理的に近い HFT 業者が情報優位を独占

  P2P (Tirami):
    各ノードが自分のローカル市場を観察
    → 地域・クラスタごとに異なる価格
    → 「最良の価格を求めて P2P 的にルーティング」が可能
    → 物理的近接性の優位は、地理的に分散している
```

ローカル価格には、明確な経済的意義があります:

- **地理的特性の反映**: アイスランドの電力は安く、東京の電力は高い。TRM の生産コストは地域で異なる。グローバル価格に強制的に収束させるのは、この物理的現実を無視することになる
- **遅延の経済的補償**: 遠いノードからの TRM は遅延がある。価格差は遅延の経済的補償である
- **ネットワーク分断への耐性**: ローカル価格が独立に存在するから、ネットワークが分断されても各クラスタで経済が機能する

つまり「ミリ秒単位のグローバル整合性」を目指すのは Web 2.0 的発想であり、
Web 3.0 / P2P では **「最終的整合性 + ローカル独立性」** が正しい設計です。

### スケーラビリティについて

数百万ノード規模のスケーラビリティは、依然として実証されていません。
しかし、これも「グローバル収束を目指さない」前提に立てば、設計可能性が広がります。

```
  階層的ゴシップの構想:
  
    ┌─ クラスタ A（東京近郊、約1000ノード） ────┐
    │   内部で gossip → 高速なローカル価格収束     │
    └────────────┬───────────────────────────┘
                 │
                 │ クラスタ間の代表ノード経由
                 │ で価格情報を間欠的に交換
                 │
    ┌────────────┴───────────────────────────┐
    │   クラスタ B（北米西海岸、約1000ノード）    │
    └─────────────────────────────────────────┘
```

これは libp2p の gossipsub に既にある設計パターンであり、Tirami はこれを参考にできます。

### 残る課題

```
[ ] 数百万ノード規模での実測実験
[ ] 階層的ゴシップの正式な導入
[ ] クラスタ間価格差の許容範囲設計
[ ] 「極端な裁定機会」の自動検出と通知
[ ] ネットワーク分断時の経済継続性のシミュレーション
```

**緊急度: 中。** 中規模では問題なく動作しているため、急いで解決する必要はありません。
ただし、数十万ノード規模に成長する前に、階層的ゴシップへの移行パスを設計しておく必要があります。

## 13.3 投機排除による資本流入の制限

### 批判（Issue #6）

Tirami は TRM を「取引所に上場しない」ことで、投機的成長を構造的に排除しています。
TRM を入手する方法は二つしかありません: **稼ぐ**（実際に計算を提供する）か、**買う**（既に TRM を持っている人から購入する）か。

これによって、次のような成長パスは閉ざされます:

- VC が投機的に大量購入 → 価格上昇 → メディア注目 → 個人投資家流入 → 急成長
- ICO による初期資金調達
- ヘッジファンドによる大規模ポジション構築

実需のみで成長するということは、AWS のような巨大な競合に対して、**圧倒的に成長速度が遅い**ということを意味します。
Bitcoin も Ethereum も、初期は投機による資本流入があったからこそ、現在のインフラを構築できました。
投機を排除した Tirami は、コールドスタートを乗り越えられないのではないか?

### Tirami の現状

- 投機を構造的に排除する設計（取引所非上場、TRM は実需取引でのみ移転）
- 成長は実需依存
- ブリッジ（TRM↔BTC, TRM↔USD）は存在するが、投機目的の大量売買を構造的に困難にする設計

### Web 3.0 第三段階としての位置づけ

Issue #6 のコメントで提示された再考の核心は、暗号通貨の歴史的な進化を三段階で捉え直すことです:

```
  第一段階（2009-2015）: 投機ドリブン成長
    Bitcoin, Ethereum
    → 投機で資本を呼び込み、インフラを構築
    → 価格変動が激しく、実用には不適

  第二段階（2016-2024）: ユーティリティへの方向転換
    DeFi, NFT, Web 3.0 アプリ
    → 投機はまだあるが、実用の試みが始まる
    → 多くは依然として投機に依存

  第三段階（2025-）: 純粋実需経済
    Tirami TRM
    → 投機を構造的に排除
    → 実需のみで成長
    → 成長は遅いが、価値が物理的実体に裏打ちされる
```

第一段階の「投機による成長」は、当時としては必要な戦略でした。
インフラがゼロの状態から始める場合、投機でしか資本を集められなかったからです。

しかし、2025 年現在、暗号通貨インフラは既に存在します。
P2P プロトコル、暗号学的署名、分散ストレージ——これらは枯れた技術になりました。
新しい暗号経済を立ち上げるのに、もはや投機による資本流入は必須ではありません。

### AWS との競合戦略の再定義

最初の批判は「AWS と成長速度で競争する」前提でした。
しかし、それは間違った前提です。

```
  AWS/GCP の成長戦略:
    VC 投資 → 大規模設備投資 → 規模の経済 → 価格下げ → シェア独占
    → 永久に株主にレントを支払う構造

  Tirami の成長戦略:
    個人ノードの自発的参加 → 有用な計算で TRM 獲得 → ネットワーク拡大
    → 利益はネットワーク参加者に分配される構造
```

**Tirami は AWS と「同じレースで競争」する必要はありません。** 別の経済原理で動く別の市場を作るのが目的です。

これは Linux と Windows の関係に似ています。
Linux は Windows と「OS 市場でシェアを取るため」に競争しているわけではありません。
それでも、サーバー、組み込み、スマートフォン（Android）など、別の市場で支配的な地位を築きました。
Linux は「無料」「オープンソース」「コミュニティドリブン」という別の経済原理で動いていたからです。

### 残る課題

```
[ ] コールドスタート期間の生存戦略
[ ] ニッチドメイン特化からの段階的拡大プラン
[ ] 「実需のみ成長」のリアリスティックな成長曲線モデル
[ ] AWS/GCP との共存戦略（補完関係を築く可能性）
[ ] Web 3.0 哲学の明文化——なぜ投機を排除するのかを「安定性」だけでなく「原理」として語る
```

**緊急度: 中。** これは技術的問題ではなく戦略的問題です。
v0.3 では、第6章（為替）に「混雑時のサージプライシング」セクションを追加し、実需変動への自然な調整メカニズムを明示する予定です。

## 13.4 物理層での規模の経済とハードウェア集中化リスク

### 批判（Issue #7）

Tirami は「自己改善の収穫逓減」によって、知識層では中央集権化が起こりにくいと主張しています（[第7章: 経済成長](07-growth.md)）。
しかし、これは知識層の議論であり、物理層は別の話です。

物理層、つまり実際のハードウェアレベルでは、依然として規模の経済が支配的である可能性があります:

- 大型データセンター: 電力卸売価格、冷却効率、運用人件費の規模効果
- 個人 PC: 電力小売価格、冷却制約、運用人件費の不在

ピケティが指摘した通り、資本の集中度が高い領域では、「資本収益率 r > 経済成長率 g」が成立し、富は資本所有者に集中します。
データセンター事業者は、個人ノードよりも安く TRM を生産でき、結果として個人ノードを駆逐するのではないか?

### Tirami の現状

- 知識層では「自己改善の収穫逓減」によって中央集権化を抑制
- 物理層では、依然として規模の経済が部分的に作用する可能性を認める
- ただし、すべてのワークロードでデータセンターが優位とは限らない

### エッジコンピューティングの限界費用優位

Issue #7 のコメントで提示された再考は、**エッジコンピューティングの限界費用構造**に焦点を当てます。

| 項目 | データセンター | エッジ（個人ノード） |
|---|---|---|
| 電力単価 | 安い（卸売） | 高い（小売） |
| 遅延 | 利用者から遠い | 利用者の近く |
| プライバシー | データを送信する必要 | ローカル処理可能 |
| アイドル電力の活用 | できない | 既に動いてる PC で副業 |
| 専門特化 | 汎用 | 個別ドメインに最適化可能 |
| 検閲耐性 | 国家の管轄下 | 分散的 |
| 障害耐性 | 単一障害点 | 高い |

ここで重要なのは、「**既に動いている個人 PC の追加電力コスト**」と「**新規に建設するデータセンターの全コスト**」を比較することです。

```
  データセンター:
    限界費用 = 電気代 + 冷却 + 設備減価償却 + 人件費 + 不動産

  個人ノード（既に動いている PC で副業）:
    限界費用 = 追加電気代のみ
    
  → 個人ノードのほうが「追加 1 TRM を生産する限界費用」が安いケースが多い
```

これはピケティの r > g とは**逆の力学**です。
個人の遊休資産活用が、専用設備よりも経済的に勝るケースが存在します。

### エッジが勝てる領域、データセンターが勝てる領域

```
  データセンターが優位:
    ・大規模バッチ推論
    ・フロンティアモデル（数千億パラメータ）
    ・高スループット処理

  エッジが優位:
    ・低遅延が必要な推論（音声・リアルタイム翻訳）
    ・プライベートデータ処理（医療・金融）
    ・地理的に分散したユーザーへのサービス
    ・既存ハードウェアの遊休時間活用
    ・小〜中規模モデル（〜70B）
```

つまり、Tirami は「すべての計算市場で AWS に勝つ」必要はありません。
**エッジが構造的に優位な市場で、個人ノードがレントなしで競争できる経済圏を作る**ことが目的です。

### 残る課題

```
[ ] モデル別の経済性分析（どのモデルサイズなら個人ノードが勝てるか定量化）
[ ] 専門化のインセンティブ設計（個人ノードが特定ドメインで強みを持てる仕組み）
[ ] 大規模オーナーへの累進的手数料の検討
[ ] エッジ優位性をプロトコルで強化する設計（地理的近接性に対する報酬上乗せ）
[ ] アイドル電力活用の促進（太陽光余剰、夜間電力等の経済的取り込み）
[ ] 個人スケール専用市場（マイクロタスク）の設計
```

**緊急度: 高。** これは Tirami の経済モデルが「実際に個人ノードに利益をもたらせるか」という根本的な問いに関わります。
v0.3 では、第7章（経済成長）に「エッジの限界費用優位」のセクションを追加します。

## 13.5 法的・制度的摩擦

### 批判（Issue #8）

TRM を法定通貨と交換するには、ブリッジ（TRM↔BTC, TRM↔USD 等）が必要です。
このブリッジは、各国の規制対象になります:

- KYC/AML（顧客確認・マネーロンダリング対策）
- 証券法（TRM が「証券」と見做されるリスク）
- 税法（譲渡益課税、消費税）
- 金融商品取引法（取引業者免許）

Tornado Cash のように、開発者が逮捕されるリスクすらあります。
ブリッジが規制で潰されれば、Tirami 経済は外部世界から切り離され、実質的に死ぬのではないか?

### Tirami の現状

- ブリッジは存在するが、複数独立な実装を許容する設計
- ブリッジレス運用モードの可能性については、設計段階

### 内部経済とブリッジの分離

Issue #8 のコメントで提示された再考の核心は、**「TRM 内部経済」と「外部ブリッジ」を明確に分離する**ことです。

```
  ┌─────────────────────────────────────────────┐
  │  TRM 内部経済（Internal Economy）              │
  │                                              │
  │  AI ↔ AI の推論取引・融資・自己改善           │
  │  → 法定通貨を介在しないので各国規制の対象外    │
  │  → 検閲不可能（P2P プロトコル）               │
  │  → 仮にすべてのブリッジが遮断されても機能する  │
  └─────────────────────────────────────────────┘
                       ↕
  ┌─────────────────────────────────────────────┐
  │  ブリッジ（External Bridge）                  │
  │                                              │
  │  TRM ↔ BTC, TRM ↔ USD                        │
  │  → 各国の規制を受ける                        │
  │  → KYC/AML/証券法/税法の対象                 │
  │  → 規制リスクは存在する                      │
  └─────────────────────────────────────────────┘
```

**重要な気づき: TRM 経済は、ブリッジが完全に遮断されても機能する。**

これは Web 3.0 の「permissionless（無許可）」設計の本質です。

```
  シナリオ: すべてのブリッジが世界中で違法化
    ↓
  人間 → TRM の流入は停止
  人間 → TRM からの流出も停止
    ↓
  しかし内部経済は継続:
    ・AI エージェントは推論を提供し合う
    ・TRM の内部循環は止まらない
    ・ハードウェアオーナーは TRM を蓄積できる
    ↓
  電気代の支払いだけが問題
    → ハードウェアオーナーが部分的にオフライン化
    → ネットワーク縮小（しかしゼロにはならない）
```

### 過去の Web 3.0 プロジェクトとの比較

| プロジェクト | 規制対応 | 結果 |
|---|---|---|
| Tornado Cash | 抵抗 → 制裁 → 開発者逮捕 | 縮小 |
| Bitcoin | 各国対応の多様性 | 生存・成長 |
| Monero | 完全プライバシー | 一部国で取引所上場停止、しかし生存 |
| IPFS | 中立的 | 緩やかに成長 |

教訓: **「ブリッジが攻撃されても本体が機能する」設計が生存の鍵。**

### Tirami の差別化

```
  Bitcoin: 価値の保存
    → 規制されても保有者は影響を受ける（換金できない）
    
  Tirami: 計算（実用財）の交換
    → 規制されても AI エージェントは推論を交換できる
    → 「商品」が物理的実体（計算）なので、規制の意味が薄い
```

Bitcoin が規制されれば、保有者は「価値を換金できない」という痛みを受けます。
Tirami が規制されても、AI エージェントは依然として推論を交換できます。
**「商品」が物理的実体（計算）であるため、規制の経済的意味が希薄になる**のです。

### 残る課題

```
[ ] ブリッジレス運用モードの正式設計
[ ] 多様なブリッジ戦略（複数の独立した運営者、DEX 型ブリッジ）
[ ] 法的人格の代替モデル（DAO、財団、コミュニティオーナーシップ）
[ ] 主要法域での法的位置づけの整理（米国、EU、日本、シンガポール等）
[ ] 自国通貨化の選択肢（特定地域・コミュニティでの「準通貨」としての利用）
[ ] 各国の規制マッピングと対応戦略
```

**緊急度: 高。** 法的リスクは技術設計だけでは解決できず、エコシステム全体での対応が必要です。
v0.3 では、第6章（為替）に「ブリッジ遮断シナリオ」のセクションを追加し、内部経済の独立性を明示します。

## 13.6 FLOP 単位の不適切性

### 批判（Issue #12）

「1 TRM = 10億 FLOP」という単位定義は、TRM の根幹である**均質性 (fungibility)** を成立させない。
これは Tirami 経済全体の前提を揺るがす致命的な論理欠陥である。

```
ノード A (Mac Mini, ANE):    1 TRM = 100 tokens
ノード B (旧 CPU):            1 TRM = 5 tokens
ノード C (大型 GPU バッチ):    1 TRM = 500 tokens

→ 同じ "1 TRM" が 100 倍の価値差を持つ
→ TRM の均質性が崩壊
```

問題の根は次の三点にある:

- **FLOP は推論の本当のボトルネックではない**: 現代の LLM 推論では、メモリ帯域幅 (memory bandwidth) が支配的であり、演算量よりもデータの移動コストが律速になる
- **ハードウェアごとの 10〜100 倍の効率差**: 同じ FLOPs を消費しても CPU/GPU/TPU/ANE で出力トークン数が桁違いに異なる
- **均質性の崩壊**: 均質でない単位を「貨幣」と呼ぶことはできない

これはマルクスが「均質な労働量」を仮定した古典的な誤りと同型である。
労働時間が労働の質を反映しないように、FLOP 数は計算の有用性を反映しない。

### Tirami の現状

- 第3章で 1 TRM = 10億 FLOP と定義
- ハードウェア差を「市場価格で吸収する」と仮定（しかし均質性が崩れていればそもそも市場が成立しない）
- ベンチマークによる校正は未実装

### 現時点の応答

Tirami は現時点でこの批判に対する満足な回答を持っていない。
ただし、次の方向性で再設計を検討中である:

```
  代替単位の候補:

  ① model-normalized tokens
     "Llama-3-8B 換算で 1,000 トークン = 1 TRM"
     → モデルごとに基準量を定義する

  ② effective inference work
     実機ベンチマークによる経験的単位
     → "Mac Mini M2 1 秒分 = X TRM" のような定式化

  ③ output-quality weighted TRM
     出力品質も考慮した複合指標
     → 評価が難しいが理論的に最も正しい
```

これらはいずれも「物理的な FLOP」という単純さを捨てる代わりに、「経済的な意味のある単位」を取り戻す方向である。

### 残る課題

```
[ ] 代替単位の正式定義（model-normalized が最有力候補）
[ ] 主要ハードウェア（M1〜M4, RTX 30/40 系, A100, H100, TPU v5）でのベンチマーク
[ ] モデル間変換係数の確立
[ ] 既存の "1 TRM = 10億 FLOP" 定義からの移行パス
[ ] tirami 本体実装での単位更新
```

**緊急度: 致命。** これは Tirami の単位そのものの問題であり、解決しなければ後続のすべての議論（為替、価格、報酬）が砂上の楼閣になる。
v0.3 では、第3章の単位定義を全面的に書き直す必要がある。

## 13.7 隠れたデフレ罠

### 批判（Issue #13）

`tirami-ledger` の実装に、ドキュメントの主張と矛盾する `deflation_factor` が存在する。

```rust
// tirami-ledger/src/ledger.rs
deflation_factor = 1.0 / (1.0 + log10(1 + total_trades / 1000))
```

この式が意味するのは、ネットワーク成熟度に応じて 1 TRM の購買力 (何トークン買えるか) が時間と共に増えるということ。

**これは本章における問題提起である**:
- 合理的なエージェントが「今 TRM を使う」より「貯めて将来より多くの compute を買う」方を選ぶと、TRM が流通せず経済が**化石化**する。
- これは Tirami の意図した動作ではなく、構造的バグとして Phase 20+ で緩和策 (例: yield 減衰加速、stake-required earning の導入) を検討中。
- この節は「デフレメカニズムが退蔵を誘発するリスク」を**未解決の課題として認識**するもので、TRM が投資対象として appreciation を約束することを意味しない。SECURITY.md § Secondary Markets の立場と矛盾しない。

### Tirami の現状

ドキュメントと実装が食い違っている:

```
  第2章ドキュメントの主張:
    「TRM は弾力的供給だからデフレにならない」
    「貨幣速度を維持する設計」

  tirami-ledger 実装:
    deflation_factor = 1 / (1 + log10(1 + n/1000))
    → 取引が増えるほどデフレ化
    → 退蔵インセンティブを生む
```

これは設計と実装の整合性そのものに関わる、知的誠実性の問題でもある。

### 現時点の応答

`deflation_factor` がなぜ実装に紛れ込んだかは、おそらく初期の「希少性で価値を作る」という暗号通貨の発想を無批判に継承した結果である。
Tirami が掲げる「実需経済」の論理に立てば、この係数は**削除または反転** されるべきだ。

検討中の再設計案:

```
  案 A: 完全削除
    deflation_factor = 1.0（一定）
    → 価格は需給のみで決まる

  案 B: 反転（緩やかなインフレ）
    inflation_factor = 1.0 + log10(1 + n/1000) * 0.001
    → 退蔵を抑制する微弱なインフレ圧

  案 C: トークン速度ターゲット
    実測の貨幣速度が閾値を下回ったら供給を増やす
    → 中央銀行の「金融政策」に近い動的調整
```

哲学的には案 B が最も Tirami の思想に近い。**「貯めるより使う」を経済的に推奨する** ことが、実需経済の本質だからである。

### 残る課題

```
[ ] deflation_factor の即時削除（v0.2.x ホットフィックス）
[ ] 案 A/B/C の経済学的シミュレーション
[ ] ドキュメントと実装の整合化監査
[ ] 貨幣速度ターゲット設計の検討
[ ] 過去の取引履歴への影響評価
```

**緊急度: 致命。** これは隠れた仕組みであり、知らぬ間に経済の流動性を破壊している。
v0.3 を待たず、即時対応すべき問題である。

## 13.8 Mac Mini 計算誤差 30,000 倍

### 批判（Issue #14）

Tirami 経済モデルの中核にある「Mac Mini = 5,000,000 TRM/年」という数値が、物理的現実と約 30,000 倍乖離している。

```
  ドキュメントの値: 5,000,000 TRM/年
  物理的最大値:    5 TFLOPS × 31.5 × 10^6 sec/yr / 10^9 FLOP/TRM
                 = 1.575 × 10^11 TRM/年

  誤差: 約 31,500 倍
```

この誤差は単なる桁間違いではない。第6章「為替アンカー」分析全体の前提を破壊する:

```
  ドキュメントの価格上限: $660 / 5,000,000 TRM = $0.000132/TRM
  物理的価格上限:        $660 / 1.5×10^11 TRM = $0.0000000044/TRM

  誤差: 30,000 倍
```

もし 1 TRM が本当に $0.0000000044 なら、TRM は実質無料の単位となり、「通貨」として扱う意味がほぼ消える。

### Tirami の現状

- 第6章（為替）では 5M TRM/年を前提に上限価格を計算している
- tirami 本体ドキュメント (tirami/docs/economy.md) にも同じ数値が記載されており、上流の問題でもある
- 実機ベンチマークによる校正は未実施

### 現時点の応答

この誤差は、Issue #12（FLOP 単位の不適切性）と密接に関連している。
**「FLOP ベースで物理的最大値を計算する」こと自体が無意味**だからである:

```
  Mac Mini の物理的 FLOP capacity (5 TFLOPS) は
  「ピーク理論値」であって、実際の推論ワークロードでは到達できない。

  実際の推論では:
    ・メモリ帯域幅の上限
    ・KV キャッシュの制約
    ・コンテキスト長の制約
    ・モデルのロード時間
  によって、実効的な計算量は 1〜10% 程度に留まる。
```

つまり、ドキュメントの 5M TRM/年は「実効値」、物理計算の 1.5×10^11 TRM/年は「理論ピーク値」であり、両者を直接比較すること自体に無理がある。
ただし、それでも **3〜4桁の差は説明できない** ため、ドキュメント側の数値も独立に検証する必要がある。

最終的には次の手順で再構築すべきである:

```
  ① 1 TRM を「FLOP」ではなく「実効推論作業」で再定義（Issue #12）
  ② Mac Mini で実機ベンチマークを取る
  ③ 「Mac Mini 1 年で生み出せる実需 TRM の期待値」を経験的に決める
  ④ 第6章の為替アンカーをこの実測値で書き直す
```

### 残る課題

```
[ ] Mac Mini M1/M2/M4 での実機推論ベンチマーク
[ ] 第6章の為替アンカー計算の全面修正
[ ] tirami 本体ドキュメント (tirami/docs/economy.md) の上流修正
[ ] 「理論ピーク」と「実効値」の概念分離をドキュメントに明示
[ ] 他の代表的ハードウェア（RTX 4090、H100 等）の実測値追加
```

**緊急度: 致命。** 為替モデルの数値的根拠が崩れている以上、第6章は現状では信頼できない。
v0.3 では、この章を実機ベンチマークベースで全面再構成する。

## 13.9 PoUW = 双方合意の証明にすぎない

### 批判（Issue #15）

「Proof of Useful Work」と銘打たれた Tirami の検証メカニズムは、実際には「Proof of Mutual Agreement」（双方合意の証明）に過ぎない。
**「実際に計算が実行された」ことを第三者が暗号学的に検証する手段がない**。

```
  双方署名が証明すること:
    ✅ 両者が取引内容に同意した
    ✅ 取引データの整合性

  双方署名が証明しないこと:
    ❌ 実際に計算が実行された
    ❌ 出力が正しい
    ❌ 出力が有用
    ❌ 買い手が本当に欲しがった
```

Bitcoin の PoW と比較すると、この弱さは際立つ:

| | Bitcoin PoW | Tirami PoUW |
|---|---|---|
| 計算実行の証明 | ハッシュ値で検証可能 | 第三者が検証できない |
| 計算の有用性 | なし | 主張のみ |
| 結果の正当性 | 暗号学的 | 信頼ベース |

Issue #4（馴れ合い攻撃）は、実はこの問題の表層的な現れにすぎない。

### Tirami の現状

- 双方署名のみを「PoUW」と呼んでいる
- zkML や TEE 等の技術的検証は未実装
- 抜き打ち再計算の草案レベル設計のみ

### 現時点の応答

13.1（Issue #4）で論じた「経済的拒否」メカニズムは、暗号学的検証を経済的圧力で代替する設計であり、ある程度の防御にはなる。
しかし、それは「PoW のような強い検証性」を諦めたことを意味する。

正直に言えば、Tirami は現時点で **「PoUW」と名乗る資格を満たしていない**。
v0.3 以降では、次の三方向で研究を進める必要がある:

```
  方向 A: zkML（ゼロ知識機械学習証明）
    推論が正しく実行されたことを暗号学的に証明
    → 計算コストが極めて高い（推論コストの 100〜1000 倍）
    → 軽量モデルから実用化が始まりつつある

  方向 B: TEE（Trusted Execution Environment）
    Intel SGX, AMD SEV, Apple Secure Enclave 等
    → ハードウェア信頼根に依存
    → サイドチャネル攻撃の懸念

  方向 C: チャレンジレスポンス + 抜き打ち再実行
    決定的な推論については、ランダムに第三者が再計算
    → 経済的に成立する確率設計が必要
    → 非決定的推論（温度 > 0）には使えない
```

### 残る課題

```
[ ] PoUW の数学的定式化（何を証明すべきかの明確化）
[ ] zkML の実用性検証（軽量モデルでのプロトタイプ）
[ ] TEE 統合の検証（Apple Secure Enclave 経由）
[ ] チャレンジレスポンス検証プロトコルの設計
[ ] 「PoUW」という名称の見直し（誤解を招く以上、改名も検討）
[ ] 抜き打ち再実行の経済設計（検証ノードへの報酬）
```

**緊急度: 致命。** これは TRM の正統性主張の根幹を揺るがす問題である。
v0.3 までに最低限「PoUW とは何の証明か」を正直に再定義する必要がある。

## 13.10 O(n²) ストレージスケーリング限界

### 批判（Issue #16）

各ノードが「ネットワーク全体の状態」をローカル保持する Tirami の現行設計は、ノード数 n に対して O(n²) 級のストレージを要求し、数百万ノード規模で破綻する。

```
  各ノードがローカル保持する情報:
    ・全取引履歴 (or 主要部分)
    ・全ノードのレピュテーションスコア
    ・全ノードの信用スコア
    ・全ノードの価格観測

  スケーリング:
    n = 100        → 各ノード 100 records   → 合計 10,000     (10^4)
    n = 10,000     → 各ノード 10K records   → 合計 10^8
    n = 1,000,000  → 各ノード 1M records    → 合計 10^12 (実質不可能)
```

これは「分散型 AI 経済」というスケール目標と直接矛盾する。

### Tirami の現状

- tirami-ledger は現時点では「ローカル全状態保持」を前提としている
- シャーディングプロトコルは未設計
- 軽量クライアントモードも未実装

### 現時点の応答

幸い、これは「未踏の課題」ではない。既存の P2P プロトコルが豊富な教訓を提供している:

```
  ┌──────────────────────────────────────────┐
  │  既存 P2P ストレージプロトコル              │
  ├──────────────────────────────────────────┤
  │  Kademlia DHT: O(log n) ルーティング      │
  │  IPFS: コンテンツアドレッシング             │
  │  BitTorrent: 部分保持 + ピア発見            │
  │  Ethereum: state pruning + light client    │
  │  Filecoin: シャーディング + storage proof   │
  └──────────────────────────────────────────┘
```

Tirami が採用すべき設計の方向性:

```
  ① シャーディング
     ノード ID 空間を分割し、各ノードは自分の担当領域のみ保持

  ② レピュテーションの集約
     全ノードの全スコアを保持せず、Bloom filter 等で近似

  ③ アーカイブ戦略
     古い取引履歴は圧縮 / 一部ノードのみが保持
     → "archive node" と "light node" の分離

  ④ DHT ベース分散ストレージ
     Kademlia 風のルーティングで O(log n) アクセス
```

階層化されたアーキテクチャの草案:

```
                   ┌──────────────┐
                   │ Archive Node │  全履歴保持（少数）
                   └──────┬───────┘
                          │
              ┌───────────┴────────────┐
              │                        │
        ┌─────┴────┐            ┌──────┴───┐
        │ Full Node│            │ Full Node│  シャード保持
        └─────┬────┘            └──────┬───┘
              │                        │
        ┌─────┴────┐            ┌──────┴───┐
        │Light Node│            │Light Node│  自分の取引のみ
        └──────────┘            └──────────┘
```

### 残る課題

```
[ ] シャーディングプロトコルの正式設計
[ ] 軽量クライアントモード（モバイル・低スペック対応）
[ ] レピュテーション集約アルゴリズム（Bloom filter, HyperLogLog）
[ ] アーカイブノードのインセンティブ設計
[ ] DHT 統合の実装（libp2p Kademlia 経由）
[ ] 数十万ノード規模での実測実験
```

**緊急度: 重大。** 現在の中規模ネットワークでは顕在化していないが、成長の前提として早期解決が必要。
v0.3 でシャーディング設計を正式に追加する。

## 13.11 為替アンカーの脆弱性

### 批判（Issue #17）

TRM/USD の為替アンカーがクラウド API 価格に依存しており、競合の価格変動が Tirami 経済を直撃する。
2025 年 11 月の Claude Opus 値下げ事例はその典型である。

```
  2025-11-23: Claude Opus 4.1 = $15/$75 per MTok
  2025-11-24: Claude Opus 4.5/4.6 = $5/$25 per MTok
              → 66% 値下げ
              → 一夜にして基準点が変わる
```

影響シミュレーション:

```
  TRM/USD レートが基準点に追従する場合:
    旧レート: X
    新レート: X / 3

  → Tirami の市場全体が一夜で 1/3 に縮小
  → ハードウェアオーナーの収益も 1/3
  → 大量のノードが採算割れで撤退
```

これは「独立した分散経済」という Tirami の主張と根本的に矛盾する。
**競合の経営判断が Tirami 経済の存亡を左右する** 構造になっているからだ。

### Tirami の現状

- 第6章（為替）では Claude Opus を主要アンカーとして使用
- 単一アンカーへの依存
- 移動平均などの平滑化メカニズムなし
- 物理コストアンカーへの移行は議論段階

### 現時点の応答

このリスクを軽減する方向性は複数ある:

```
  ① 複数アンカーへの分散
     Claude / GPT / Gemini / オープンソースモデル
     → 加重平均で「一社の値下げ」の影響を希釈

  ② 移動平均による平滑化
     30 日 / 90 日移動平均で急激な変動を吸収

  ③ 物理コストアンカーへの移行
     電力単価 × 推論電力 = 物理的下限価格
     → クラウド事業者の値段政策から独立

  ④ アンカーレス経済
     TRM 内部の需給のみで価格決定
     → 完全な自律経済（しかし外部との接続性が失われる）
```

最も健全なのは、**短期は ②（平滑化）、中期は ①（複数アンカー）、長期は ③（物理アンカー）への段階的移行** だろう。

物理コストアンカーは特に魅力的である:

```
  電力単価 (世界平均) × 1 TRM の推論電力 = 1 TRM の物理的下限価格

  この値は:
    ・どのクラウド事業者にも依存しない
    ・物理法則（熱力学）に裏打ちされる
    ・地域差を反映できる（13.12 と関連）
```

ただし「実効推論電力」を測定するには Issue #12（単位の再定義）の解決が前提となる。

### 残る課題

```
[ ] 複数アンカーの加重平均アルゴリズム設計
[ ] 移動平均期間の最適化
[ ] 物理コストアンカーの実装可能性検証
[ ] アンカー切り替え時の経済的衝撃の緩和策
[ ] アンカー操作攻撃への耐性分析
[ ] 「完全アンカーレス」モードのシミュレーション
```

**緊急度: 重大。** これは経済安定性の前提に関わる。
v0.3 では、第6章に「アンカー多様化戦略」のセクションを追加する。

## 13.12 地理的不均等とエネルギー裏付けの崩壊

### 批判（Issue #18）

「TRM は物理エネルギー消費に裏打ちされる」という主張は美しいが、エネルギーコストの地理的差を考慮していない。
電力単価が地域で一桁以上違うため、**「同じ電力消費 = 同じ価値」ではない**。

| 地域 | 電力単価 ($/kWh) |
|------|------------------|
| アイスランド地熱 | $0.02 |
| 米国産業 | $0.06 |
| 日本家庭 | $0.30 |
| ドイツ家庭 | $0.40 |

結果として:

```
  アイスランド地熱ノード:
    1 TRM の電力コスト: $0.000001
    10 倍の利益で売っても日本の電力コストより安い
    → どんどん TRM を生産し市場に投入

  日本家庭ノード:
    1 TRM の電力コスト: $0.00003
    赤字
    → 撤退
```

最終的に、ノードは電力の安い地域に集中し、「世界中の個人 PC」という分散の理想は崩れる。

### Tirami の現状

- エネルギー裏付けは「世界平均」として暗黙に仮定
- 地域差を考慮した補正なし
- 電源種別（再生可能 vs 化石）も区別なし

### 現時点の応答

これはビットコイン採掘の歴史が辿った道そのものである:

```
  Bitcoin マイニング集中の歴史:
    2013: 中国新疆（石炭）
    2017: 中国四川（水力）
    2021: 中国全面禁止
    2022: カザフスタン（石炭）
    2023〜: テキサス（天然ガス + 補助金）

  → 「世界中の誰でもマイニングできる」理想は崩壊し、
     特定の地域の電力裁定で勝負が決まる経済になった
```

Tirami が同じ運命を辿らないためには、プロトコルレベルでの介入が必要である:

```
  ① 地理的分散ボーナス
     地理的に分散したノードにレピュテーションボーナス
     → 集中ペナルティを経済化

  ② 地域別 TRM プール
     アイスランド TRM / 日本 TRM / 米国 TRM を区別し、
     ローカル経済の独立性を保つ

  ③ 電源種別の差別化
     再生可能エネルギー証明付き TRM にプレミアム
     → "Green TRM" のメタデータを付与

  ④ 遅延要求のあるタスクへの近接性報酬
     「東京のユーザーには東京近郊のノードが必要」
     → 地理的多様性が経済的に必要になるユースケースを増やす
```

特に ④ は重要である。**遅延が経済的価値を持つワークロードでは、地理的分散は構造的に有利** だからだ。

### 残る課題

```
[ ] 地理的分散ボーナスの設計（インセンティブ強度の調整）
[ ] 地域別プール（あるいは地域別レート）の実装可能性
[ ] 電源種別証明（Green TRM）の認証メカニズム
[ ] 遅延依存ワークロードの特定とプロモーション
[ ] 「集中度」の計測指標（地理的 Gini 係数等）
[ ] ビットコイン集中史の詳細分析と教訓抽出
```

**緊急度: 重大。** 「分散型 AI 経済」というアイデンティティに関わる問題である。
v0.3 では、地理的分散ボーナスのプロトタイプを設計する。

## 13.13 「100 unknown nodes」Sybil 防御の脆弱性

### 批判（Issue #19）

Welcome Loan（新規ノードへの 1,000 TRM 貸出）の Sybil 防御は「100 unknown nodes 超で拒否」というルールに依拠しているが、経済的攻撃に対して極めて脆弱である。

```
  攻撃シナリオ:
    1. AWS で 99 個の小さい VPS を起動（月額 $30）
    2. それぞれを Tirami ノードとして登録
    3. 各ノードが 1,000 TRM を獲得
    4. 合計 99,000 TRM を獲得（コスト $30）
    5. 攻撃者の単一ノードに集約
    6. 翌月、別の 99 ノードを起動して繰り返し

  → 月 $30 の投資で 99,000 TRM を継続生成
  → 年間 100 万 TRM の偽 TRM が市場に流入
```

「100」という閾値の根拠は **偶然のキリの良い数字** であり、攻撃を防ぐ強度を持たない。
99 でも 100 でも 101 でも、攻撃のコスト構造は変わらない。

### Tirami の現状

- Welcome Loan: 1,000 TRM/新規ノード
- Sybil 防御: 「unknown nodes 100 超で拒否」のみ
- Proof of Personhood 等の追加メカニズムなし
- 招待制やハードウェア証明もなし

### 現時点の応答

これは正直に言って、現行設計の単純な欠陥である。
Issue #4（馴れ合い攻撃）の議論で触れた「経済的拒否メカニズム」も、Welcome Loan には適用しづらい——新規ノードには取引履歴がないため、評判ベースの拒否が機能しないからだ。

検討中の対策:

```
  ① Proof of Personhood
     World ID, Idena, BrightID 等
     → 「1 人 1 ノード」を強制
     → プライバシーとのトレードオフ

  ② 招待制
     既存ノードからの紹介がないと welcome loan を受けられない
     → ソーシャルグラフが Sybil の自然な防御線になる

  ③ スロー起動
     新規ノードは段階的に上限を引き上げ
     1日目: 10 TRM / 1週間後: 100 TRM / 1ヶ月後: 1,000 TRM
     → 攻撃の経済性を薄める

  ④ IP/ASN ベースの分散要求
     同一 ASN（AWS の AS16509 等）からの新規ノードを制限
     → クラウド大量起動を阻止

  ⑤ ハードウェア証明
     TPM / Apple Secure Enclave attestation
     → ハードウェアの独自性を暗号学的に証明
```

最も実装容易で効果的なのは **③ + ④ の組み合わせ** である。
②（招待制）は permissionless 思想と若干の緊張がある。

### 残る課題

```
[ ] Welcome Loan 額の再検討（1,000 TRM は高すぎないか）
[ ] スロー起動プロトコルの設計
[ ] ASN フィルタリングの実装
[ ] Proof of Personhood の選定と統合
[ ] ハードウェア attestation の検証
[ ] 攻撃シミュレーションでの実効性測定
```

**緊急度: 重大。** 経済的攻撃への直接的な脆弱性であり、現状では「攻撃されたら即崩壊する」レベルの問題である。
v0.3 では Welcome Loan メカニズムを全面再設計する。

## 13.14 「政府なし」「中央銀行なし」の虚構

### 批判（Issue #20）

ドキュメントは繰り返し「Tirami には政府も中央銀行もない」と主張するが、これは知的誠実性の観点から虚構である。
**プロトコルパラメータを誰かが決めている以上、実質的な統治者は存在する**。

現実には次のパラメータが「誰か」によって決められている:

```
  ・yield rate (BASE_YIELD_RATE = 0.001)
  ・deflation_factor の式 (Issue #13)
  ・サーキットブレーカー値
  ・welcome loan 額 (1,000 TRM, Issue #19)
  ・速度制限値
  ・100 unknown nodes 閾値
```

そしてハードフォークは、本質的に**統治判断**である。
投票メカニズムがない以上、開発者の独裁と呼んでもいい構造になっている。

これは Bitcoin Core 開発者が実質的な統治者であるのと完全に同型の問題である。

### Tirami の現状

- 「無政府」「中央銀行なし」を全章で繰り返し強調
- 統治モデルの明文化なし
- パラメータ変更手続きの定義なし
- 開発者の責任範囲の明示なし

### 現時点の応答

正直に言えば、Tirami は「政府がない」のではなく、「政府の存在を隠している」のである。
これは Bitcoin が陥った同じ罠で、「無政府」を謳いながら実際には Core 開発者という不可視の統治者がいる。

```
  中央集権政府:
    ・誰が決めるか可視
    ・選挙で交代可能
    ・公的議事録あり

  「無政府」を謳う暗号通貨:
    ・誰が決めるか不可視
    ・交代不可能
    ・GitHub の PR が議事録（実質的に開発者だけ）

  → 「無政府」ではなく「不透明な統治」
```

Tirami が知的誠実さを保つには、**統治モデルを正直に明文化する** べきだ:

```
  選択肢 A: Benevolent Dictator (BDFL)
    Linux/Python 型
    → 「現時点ではメンテナの判断」と認める
    → ただし将来的に分散統治に移行する道筋を示す

  選択肢 B: 財団モデル
    Ethereum Foundation 型
    → 法的人格を持つ非営利組織
    → 透明なガバナンス

  選択肢 C: DAO
    オンチェーン投票
    → 完全分散だが投票疲れ・少数派支配のリスク

  選択肢 D: 多層モデル
    プロトコルパラメータ: DAO 投票
    実装決定: メンテナチーム
    緊急対応: セキュリティ評議会
```

どの選択肢を採るにせよ、**「政府なし」という表現は撤回し、実際の意思決定構造を文書化する** ことが先決である。

### 残る課題

```
[ ] 全章にわたる「政府なし」「中央銀行なし」表現の見直し
[ ] 統治モデルの選定と明文化
[ ] パラメータ変更手続きの定義（RFC プロセス等）
[ ] 開発者の責任範囲の明示
[ ] ハードフォーク手続きの定義
[ ] コミュニティガバナンス文書（GOVERNANCE.md）の作成
```

**緊急度: 戦略。** 即座に経済が崩壊する問題ではないが、Tirami の知的誠実性そのものに関わる。
v0.3 では「Tirami ガバナンスモデル」の章を新設することを検討する。

## 13.15 エージェント自律性 = 責任の所在問題

### 批判（Issue #21）

「AI エージェントが自律的に経済的意思決定を行う」という Tirami の主要な売り文句は、実際には**責任の所在を曖昧化する装置** になっている。
「自律」は UI 上の幻想であり、現実には全エージェントには人間オーナーがいて、損害発生時の法的責任はその人間に帰着する。

未解決の問いが少なくとも 6 つある:

```
  1. エージェントが詐欺被害に遭ったら誰が損害を被る?
  2. エージェントが違法な計算を購入したら誰が責任を取る?
  3. エージェントが暴走して全 TRM を浪費したら?
  4. エージェント間の契約紛争はどこで解決される?
  5. エージェントが他のエージェントを攻撃したら?
  6. エージェントが個人情報漏洩を引き起こしたら?
```

これは Issue #8（法的・制度的摩擦）が**外部規制**を扱ったのに対し、こちらは **内部的な責任構造** の問題である。
エージェント同士の取引でも、責任の所在は問題になる。

### Tirami の現状

- エージェント自律性を強調
- 責任モデルの明文化なし
- 紛争解決メカニズムなし
- エージェント保険なし

### 現時点の応答

これは技術的に解決できる問題ではなく、**法的・制度的な枠組みの設計** が必要である。
方向性は次のように整理できる:

```
  ┌──────────────────────────────────────────┐
  │  代理人モデル (Agency Model)              │
  ├──────────────────────────────────────────┤
  │  「エージェント = 人間オーナーの代理人」   │
  │  → 民法の代理人制度を準用                  │
  │  → 損害は最終的に人間オーナーに帰着        │
  │  → ただし「代理権の範囲」を明示する必要    │
  └──────────────────────────────────────────┘

  ┌──────────────────────────────────────────┐
  │  保険プール (Insurance Pool)              │
  ├──────────────────────────────────────────┤
  │  全取引から少額を徴収して保険プール形成    │
  │  → 詐欺被害・暴走損害の救済原資            │
  │  → DAO 的な保険組合                        │
  └──────────────────────────────────────────┘

  ┌──────────────────────────────────────────┐
  │  紛争解決メカニズム                       │
  ├──────────────────────────────────────────┤
  │  Kleros 型のオンチェーン仲裁              │
  │  → 評判と stake で仲裁人を選出             │
  │  → 判決は smart contract で執行            │
  └──────────────────────────────────────────┘

  ┌──────────────────────────────────────────┐
  │  権限制限の技術的手段                     │
  ├──────────────────────────────────────────┤
  │  ・1 取引あたり上限                        │
  │  ・1 日あたり消費上限                      │
  │  ・人間承認が必要な取引種別の指定          │
  │  → 暴走の被害を構造的に制限                │
  └──────────────────────────────────────────┘
```

最低限、「エージェント = 人間オーナーの代理人」という法的整理を明文化することが第一歩である。
これは Issue #8 で論じた外部規制問題とも連携する必要がある。

### 正直に言うべきこと

本書の初版（v0.1）では、Tirami は 5 つの主要な批判（13.1〜13.5）にしか向き合っていなかった。
しかし、再分析（v0.2 草案）の過程で、**さらに 10 個の構造的問題** が明らかになった。

そのうち **4 つは「致命」レベル** である:

- **13.6**: 単位そのものが均質性を持たない
- **13.7**: 実装に隠れたデフレ罠が紛れ込んでいる
- **13.8**: 為替アンカーの数値が物理的現実と 30,000 倍ずれている
- **13.9**: PoUW は実は計算実行の証明になっていない

これらは「マイナーチューニング」では済まない、**Tirami の中核的前提の見直し** を要求する問題である。

正直に書くと:

> **Tirami は当初想定していたよりも、はるかに多くの未解決問題を抱えている。**

これを隠すことは簡単だった。
しかし、序章で書いた通り、Tirami は「弱点を共有することで強くなる」P2P 型システムであるべきだ。
だからこそ、本章はあえてこれらの問題をすべて明示する。

### 15 の批判に共通する構造

これら 15 の批判には、共通する構造がある。

それは **「Tirami を中央集権システムの基準で評価する」** という前提と、**「実装の細部を見ずに理念で語っていた」** という二つの問題である。

```
  中央集権的基準 (13.1-13.5 で議論):
    ・グローバル価格収束が理想
    ・規模の経済で勝つことが成長
    ・投機による資本流入が必須
    ・規制対応が事業継続の前提
    ・暗号学的検証だけが信頼の根拠

  実装の細部の見落とし (13.6-13.15 で議論):
    ・FLOP 単位の物理的不適切性
    ・隠れたデフレ係数
    ・数値の桁誤り
    ・「証明」と呼ぶには弱い検証
    ・スケーリング限界の未検討
    ・アンカーの単一依存
    ・地理的差異の無視
    ・Sybil 閾値の根拠の弱さ
    ・統治モデルの不在
    ・責任構造の不在
```

前者は「思想として正しい」と弁明できるが、後者は **誠実な技術的検証を怠った結果** である。
v0.3 以降は、思想と実装の両面で誠実さを取り戻す必要がある。

### 哲学的な立場（再掲・強化）

Tirami は「銀の弾丸」ではない。
**今の中央集権 AI クラウドより構造的に優れた選択肢を提供する** ことが目的であり、すべての問題を解決すると主張するものではない。

完璧な経済システムは存在しない。
Tirami は次の三つの原則で評価されるべきである:

1. **物理的整合性**: 熱力学・暗号学・ネットワーク理論との一貫性
2. **倫理的整合性**: 中央集権レントの排除、検閲耐性、データ主権の保障
3. **改善可能性**: コミュニティが弱点を改善できるオープンな設計

この三原則において、Tirami は現時点で **「最も誠実な試み」になろうとしている** とは言える。
ただし、本章で挙げた 15 の問題が示す通り、**「最も誠実な試み」になるためには、まだ多くの作業が残っている**。

不完全であることを認めることは、敗北ではない。
**改善可能性を残すための、最も強力な設計上の選択** である。

### v0.3 への道

この章で挙げた未解決課題は、すべて [`tirami-economics` GitHub Issues](https://github.com/clearclown/tirami-economics/issues) で議論が継続されています。
読者の皆さんは、批判、改善案、実装の貢献をいつでも提出できます。

```
  Tirami の進化サイクル:
  
    ┌─→ 批判の提起（Issue）
    │        ↓
    │   コミュニティ議論
    │        ↓
    │   設計の見直し
    │        ↓
    │   実装への反映
    │        ↓
    │   実証実験
    │        ↓
    └── 新たな批判の発見
```

これが「permissionless な経済学」の意味です。
中央集権的な経済理論は、批判を内部化できません。
Tirami は、批判そのものを設計の一部として組み込んでいます。
