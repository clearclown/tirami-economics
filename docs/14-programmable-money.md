## なぜこの章が必要か

Tirami の経済設計を説明すると、ほぼ必ず次の二つの質問を受けます。

1. 「それって JPYC みたいなものですか?」
2. 「なぜブロックチェーンに載せないんですか?」

どちらも真っ当な質問です。プログラマブルマネー (programmable money) は近年急速に実用化が進んでおり、日本では JPYC、世界では USDC や DAI が広く使われています。ブロックチェーンは「Web 3.0 の価値交換インフラ」として広く認知されており、「新しい通貨 = ブロックチェーン」という図式が定着しつつあります。

この図式のなかで、Tirami の「TRM は off-chain で、ブロックチェーンに載らない」という設計は異質に見えます。しかし、それは怠慢でも無知でもなく、**高頻度・低額・AI エージェント向けの計算経済という用途に最適化した意図的な設計** です。

この章では、その設計判断の内実を明らかにします。

## 14.1 そもそもプログラマブルマネーとは

### 14.1.1 通貨の進化という文脈

人類の通貨は、大まかに次のような段階を経て進化してきました。

```
  商品貨幣        法定通貨         電子マネー       プログラマブルマネー
  (貝殻・金)  →  (紙幣・硬貨)  →  (Suica, PayPay)  →  (JPYC, USDC, DAI)
      │              │                │                    │
   物が価値     国家が価値         企業が価値         コードが価値
   を担保       を担保             を担保             を担保
```

プログラマブルマネーとは、大雑把に言えば「**コードで条件付きの支払いを実行できる通貨**」です。

従来の電子マネーは「残高」と「送金」しかできません。Suica は 1,000 円を送れますが、「相手が商品を届けたら自動で送る」といった条件付きの送金はできません。そこには常に人間(または中央システム)の介在が必要です。

プログラマブルマネーでは、送金条件そのものをコード(スマートコントラクト)として表現できます。

### 14.1.2 スマートコントラクトの基本

スマートコントラクトを一言で言えば「**ブロックチェーン上で自動実行されるプログラム**」です。具体例を見てみましょう。

```javascript
// エスクロー付きの支払い契約
function releasePayment(goodsDelivered) {
    if (goodsDelivered) {
        transfer(seller, 1000 * JPYC);
    } else {
        refund(buyer, 1000 * JPYC);
    }
}
```

ポイントは三つです。

1. **人間の仲介なしに実行される**: コードがそのまま取引を執行する
2. **コードがイミュータブル**: 一度デプロイされたら、改ざんできない
3. **実行結果が公開される**: すべての取引がブロックチェーン上に残る

これが Ethereum のようなプラットフォームを「世界のコンピュータ」と呼ぶ所以です。

### 14.1.3 なぜプログラマブルマネーが重要か

プログラマブルマネーの応用領域は広く、代表的なものを挙げると:

- **DeFi (Decentralized Finance)**: 銀行を介さない貸借・取引所・保険
- **DAO (Decentralized Autonomous Organization)**: コードで運営される組織
- **自動化された B2B 決済**: 納品確認と同時に自動支払い
- **AI エージェントとの親和性**: 人間を介さずエージェントが自律取引できる

最後の点は Tirami と特に関係が深いです。AI エージェントが自律的に取引するためには、「人間の承認を待つ通貨」では間に合いません。プログラマブルマネー的な発想——**コードで動く通貨**——が不可欠なのです。

ただし、「プログラマブルマネー = ブロックチェーン上のマネー」と直結させるのは早計です。後述するように、Tirami は「プログラマブルなマネー」であることを維持しつつ、中核部分はブロックチェーンに載せないという設計を取ります。

## 14.2 主要なプログラマブルマネーの整理

プログラマブルマネーの代表的な 4 種類を順番に見ていきます。

### 14.2.1 JPYC (日本円ペッグ)

| 項目 | 内容 |
|------|------|
| 発行主体 | JPYC 株式会社 |
| ペッグ | 1 JPYC = 1 円 |
| 主なチェーン | Ethereum, Polygon, Shiden, Astar, Avalanche |
| 規制根拠 | 資金決済法 (前払式支払手段、現在は電子決済手段への移行準備) |
| 特徴 | 日本初の本格的な規制下ステーブルコイン |

JPYC の本質は「**既存の円を Web3 空間に持ち込むこと**」です。1 円を JPYC に換えると、その JPYC は Ethereum や Polygon の上で自由に送金・スマートコントラクト呼び出しに使えます。使い終わったら 1 円に戻す(償還する)ことができます。

ここで重要なのは、JPYC 自体は新しい価値を生み出していない点です。あくまで**既存の円に 1:1 で結びついたラッパー**であり、円の供給量が変わらない限り JPYC の供給量も変わりません。

### 14.2.2 USDC (米ドルペッグ)

| 項目 | 内容 |
|------|------|
| 発行主体 | Circle 社 (Coinbase 提携) |
| ペッグ | 1 USDC = 1 USD |
| 準備金 | 米国債 + 銀行預金 (月次監査レポート公開) |
| 主なチェーン | Ethereum, Solana, Avalanche, Base ほか多数 |
| 規制根拠 | 米国の MSB (Money Services Business) ライセンス |

USDC はステーブルコイン市場で USDT (Tether) に次ぐ第 2 位の流通量を持ちます。発行体の Circle は米国規制下で準備金を開示しており、「規制下のステーブルコイン」の代表例と見なされています。構造は JPYC とほぼ同じで、「米ドルを Web3 空間に持ち込むラッパー」です。

### 14.2.3 DAI (分散型 USD ペッグ)

| 項目 | 内容 |
|------|------|
| 発行主体 | MakerDAO (分散自律組織) |
| ペッグ | 1 DAI ≈ 1 USD |
| 担保 | ETH / WBTC / USDC 等の暗号資産 (過剰担保) |
| 主なチェーン | Ethereum |
| 規制根拠 | 法的グレー (明示的に規制下にない) |

DAI は JPYC / USDC と大きく異なります。発行体が「会社」ではなく「DAO」であり、準備金が法定通貨ではなく**暗号資産の過剰担保** (例: 150 USD 相当の ETH を預けて 100 DAI を借りる) です。理論的には中央発行体なしに USD ペッグを維持できる、という野心的な設計ですが、実運用上は USDC を相当量担保に入れており、「完全に分散」とは言い切れない面があります。

### 14.2.4 CBDC (中央銀行デジタル通貨)

| 項目 | 内容 |
|------|------|
| 発行主体 | 中央銀行 (例: 中国人民銀行、欧州中央銀行) |
| ペッグ | 法定通貨と 1:1 |
| チェーン | 専用 DLT (ほとんどパーミッションド型) |
| 規制根拠 | 国家の通貨主権そのもの |
| 特徴 | 究極的な中央集権型プログラマブルマネー |

CBDC (Central Bank Digital Currency) は、中央銀行が直接発行するデジタル通貨です。デジタル人民元 (e-CNY) はすでに実用段階にあり、ECB のデジタルユーロも計画段階です。

CBDC は「プログラマブル」と言われますが、そのプログラマビリティは**中央銀行が定義する** ものであり、市民が自由にスマートコントラクトを書けるわけではありません。たとえば「このお金は食料品にしか使えない」「3 ヶ月で期限切れ」といった制約を中央銀行が付与できる——これは通貨主権の究極形態であり、同時に強力な監視・統制手段でもあります。

### 14.2.5 比較表 (Tirami TRM を含む)

5 種類をまとめて比較すると、次のようになります。

| 次元 | JPYC | USDC | DAI | CBDC | **Tirami TRM** |
|------|------|------|-----|------|-------------|
| 裏付け | 円 | USD | ETH 等 | 法定通貨 | **計算 (FLOP)** |
| 発行主体 | JPYC 社 | Circle 社 | MakerDAO | 中央銀行 | **計算実行者本人** |
| 目的 | 円のデジタル化 | USD のデジタル化 | 分散型 USD | デジタル中央銀行 | **計算の貨幣化** |
| チェーン | Ethereum 他 | Ethereum 他 | Ethereum | 専用 DLT | **なし (off-chain)** |
| ガス代 | あり | あり | あり | 不明 (低廉想定) | **なし** |
| 規制 | 資金決済法 | MSB | グレー | 国家管轄 | **未規定** |
| 投機 | 制限あり | 制限あり | 可 | 不可 | **構造的に非推奨 (プロトコルは投機的保持を奨励しない)** |
| 典型ユーザー | 人間 | 人間 | DeFi ユーザー | 市民 | **AI エージェント** |
| 1 取引の金額規模 | 数百〜数万円 | 数百〜数万 USD | 数百〜数万 USD | 日用品 | **0.0001〜数 USD** |
| 確定時間 | 秒〜分 | 秒〜分 | 秒〜分 | 秒 | **ミリ秒** |

この表を眺めると、Tirami TRM はほぼすべての軸で他 4 者と異質であることが分かります。これは偶然ではなく、**Tirami が解こうとしている問題が根本的に違う** から生じる差異です。

## 14.3 JPYC と Tirami TRM の根本的な違い

比較表の全体像を掴んだうえで、特によく混同される JPYC と TRM の違いを詳しく見ます。

### 14.3.1 「裏付けの方向」が逆

両者の最も本質的な違いは、**価値の流れる方向** です。

```
  JPYC の発想:
    既存通貨 (円) を Web3 に載せる
         ↓
    既存経済の延長 (円 → JPYC → 円)
         ↓
    目的: 円の使い勝手を広げる

  Tirami TRM の発想:
    物理計算そのものを新しい通貨にする
         ↓
    新しい経済の創造 (計算 → TRM → 計算)
         ↓
    目的: 計算という新しい価値の単位を作る
```

JPYC は「上から下へ」流れる設計です。既存の円という価値を、デジタル空間にコピーして使いやすくする。TRM は「下から上へ」湧き上がる設計です。物理的な計算を実行するという行為自体から、貨幣が発生する。

### 14.3.2 取引フローの違い

典型的な取引フローを並べてみましょう。

```
  JPYC の取引:
    [円] → 入金 → [JPYC 発行] → 送金 → [受取人の JPYC] → 償還 → [円]
       ↑                                                       ↑
       既存経済                                              既存経済

  Tirami TRM の取引:
    [計算実行] → 双方署名 → [TRM 発行] → 送金 → [受取人の TRM] → 計算購入 → [計算実行]
          ↑                                                              ↑
          物理世界                                                     物理世界
```

JPYC は「円と円の間の通路」として機能します。TRM は「計算と計算の間の通路」として機能します。どちらが優れているという話ではなく、**扱っている価値がそもそも違う** のです。

### 14.3.3 投機耐性の違い

JPYC や USDC は 1:1 ペッグされているため、価格は安定しています。しかし、これは「意図した設計」ではなく「担保価値によって強制された安定」です。担保が毀損すれば(たとえば Circle の銀行破綻など)、ペッグは崩壊します。

Tirami TRM は、そもそも「固定ペッグ」を持ちません。TRM の価値は物理計算市場の需給で決まります。しかし、[第3章](03-supply-demand.md) や [第10章](10-pouw.md) で議論したように、TRM の発行は物理計算の実行とハードに結びついているため、**純粋な投機による価格変動は構造的に起こりにくい** 設計になっています。

### 14.3.4 競合ではなく補完

以上を踏まえると、重要な結論が得られます。

**JPYC と Tirami TRM は競合しない。** それぞれ別のレイヤーで動作する別の通貨です。

- JPYC は **人間の経済を Web3 に持ち込む** 通貨
- TRM は **AI エージェントの経済を物理世界に接続する** 通貨

両者は共存できます。というより、[§14.6](#146-jpyc-との具体的な統合パターン) で見るように、**接続されるべき** です。人間が AI エージェントに予算を渡すとき、JPYC → TRM の交換が必要になります。エージェントが使い切れなかった予算を人間に返すとき、TRM → JPYC の交換が必要になります。

## 14.4 なぜ TRM はブロックチェーンに載っていないのか

さて、核心の問いに入ります。「プログラマブルマネーの主要な実装はブロックチェーン上にある。なぜ Tirami TRM はブロックチェーンに載っていないのか?」

答えは一言で言えば「**載せると壊れるから**」です。以下、5 つのデメリットを順に見ます。

### 14.4.1 デメリット 1: ガス代の経済的破綻

Tirami TRM の典型的な取引額を考えてみます。LLM 推論 1 回のコストは、モデルと長さによりますが、おおよそ以下の規模です。

```
  小さな推論 (短文、軽量モデル)   : 0.00001 USD 〜 0.0001 USD
  中程度の推論 (GPT-4 相当)       : 0.001 USD 〜 0.01 USD
  大きな推論 (長文、重量モデル)   : 0.05 USD 〜 0.5 USD
```

一方、Ethereum のガス代は時期により激しく変動しますが、典型的には:

```
  Ethereum (L1)     : 2 USD 〜 50 USD (混雑時は 100 USD 超)
  Polygon PoS       : 0.01 USD 〜 0.1 USD
  Optimism / Base   : 0.05 USD 〜 0.5 USD
  Solana            : 0.0001 USD 〜 0.001 USD
```

ここで致命的な問題が見えます。小さな推論 1 回(たとえば 0.0001 USD)を Ethereum に載せると、**ガス代が取引額の 10,000 倍以上** になります。Polygon でも 100〜1,000 倍です。もっとも安価な Solana ですら、小さな推論ではガス代が取引額と同等になります。

```
  取引額    : ▏                        (0.0001 USD)
  Ethereum  : ████████████████████████ (5 USD = 取引額の 50,000 倍)
  Polygon   : █████                    (0.05 USD = 取引額の 500 倍)
  Solana    : ▏                        (0.0005 USD = 取引額の 5 倍)
```

1 取引あたり数十倍のオーバーヘッドがかかる経済は、**そもそも経済として成立しません**。これは技術的な未熟ではなく、「ブロックチェーンに書き込むこと自体のコスト」という構造的な問題です。

### 14.4.2 デメリット 2: 投機の流入

もし TRM が Ethereum 上の ERC-20 トークンだったら、何が起きるでしょうか。

```
  Step 1: TRM が Ethereum にデプロイされる
  Step 2: 誰かが Uniswap に TRM/USDC ペアを作る
  Step 3: 投機家が流入
          「この AI 経済は伸びる! TRM を買い占めよう」
  Step 4: TRM の価格が需給ではなく期待で動き始める
  Step 5: 計算実行者は TRM を受け取るより市場で売る方が儲かる
  Step 6: 本来の「計算の対価」という意味が崩壊
```

これは仮想の話ではなく、ほぼすべての「ユーティリティトークン」が辿った道です。Filecoin、Helium、Render、Bittensor ——本来の用途(ストレージ、ワイヤレス、レンダリング、AI)より、投機の方が主な取引動機になっています。

Tirami の設計原則の一つは「**投機を構造的に排除する**」ことです ([第10章](10-pouw.md))。DEX 上場は、この原則を根本から崩します。off-chain であることは、そもそも DEX 上場という経路を絶つための措置でもあります。

### 14.4.3 デメリット 3: 確定時間の悪化

AI エージェントは人間より圧倒的に速く判断します。1 秒間に数十〜数百の推論を発注することも珍しくありません。この速度に追いつくには、取引確定がミリ秒単位で完了する必要があります。

各チェーンの確定時間を比較すると:

```
  Tirami (off-chain)  : ~ 1 ms          ████
  Solana             : ~ 400 ms        ████████████████████
  Polygon            : ~ 2 sec         ██████████████████████████████████████
  Ethereum           : ~ 12 sec + 確定待ち (数分)  ████████████████████████████████████████████████████
  Bitcoin            : ~ 10 分 + 確定待ち (60 分)  (表示不能)
```

Ethereum の 12 秒は、AI エージェントの視点では「永遠」です。1 秒に数百の取引をしたいエージェントが、1 取引につき 12 秒待つことは不可能です。

### 14.4.4 デメリット 4: スループット限界

Tirami が想定するスケールは「**100 万台以上のエージェントが、それぞれ秒間数取引する**」という世界です。これを TPS (Transactions Per Second) に換算すると、少なくとも 100 万〜1000 万 TPS が必要です。

現状の各チェーンの TPS は:

```
  Ethereum (L1)    :       15 〜      30 TPS
  Polygon PoS      :       65 〜     7,000 TPS
  Solana           :    3,000 〜    65,000 TPS (公称)
  Tirami 想定       :1,000,000 〜10,000,000 TPS
```

最も高速な Solana ですら、桁が 2〜3 足りません。ブロックチェーンはその設計上、「全ノードが全取引を検証する」ため、スループットを上げるのが本質的に難しいのです。Tirami は「各取引を関係ノードだけが検証する」off-chain アプローチによって、この制約を回避しています。

### 14.4.5 デメリット 5: `tirami/CLAUDE.md` の設計原則違反

最後は工学的な問題ではなく、設計思想の問題です。`tirami` リポジトリの `CLAUDE.md` には明示的に次のように書かれています。

> **No blockchain in the core.**
> The core TRM economy runs off-chain with pairwise-signed transactions and gossip.

これは「とりあえず off-chain にしておいて、あとで on-chain に載せ替える」という留保ではなく、**off-chain こそが Tirami の本質** だという宣言です。もし TRM を Ethereum に載せたら、それはもはや Tirami ではありません。ここまで述べた 4 つのデメリットが同時に襲いかかり、Tirami の核心価値(高頻度・低額・ミリ秒確定・ガス代ゼロ)が完全に失われます。

### 14.4.6 数値比較表

まとめとして、off-chain と主要チェーンを並べた比較表を示します。

| 指標 | Tirami (off-chain) | Ethereum L1 | Polygon PoS | Solana |
|------|------------------|-------------|-------------|--------|
| 1 取引のコスト | **ほぼゼロ (署名のみ)** | 2〜50 USD | 0.01〜0.1 USD | 0.0001〜0.001 USD |
| 確定時間 | **~1 ms** | 12 秒 + 確定 | ~2 秒 | ~400 ms |
| TPS (現実値) | **10^6 〜 10^7** | 15〜30 | 65〜7,000 | 3,000〜65,000 |
| 小額取引 (<0.01 USD) | **可能** | 不可能 | ぎりぎり | 可能 |
| 投機流入リスク | **構造的になし** | 高 | 高 | 高 |
| 監査証跡 | なし (単独では) | 完全 | 完全 | 完全 |
| DAO 統治・スマコン | なし (単独では) | 最適 | 可 | 可 |

この表から明らかなように、off-chain は「監査証跡」と「スマコン」の 2 点で不利です。これは本物の欠点です。ではどうするか——その答えが次節のハイブリッド設計です。

## 14.5 ハイブリッド 3 層アーキテクチャ

### 14.5.1 全体像

Tirami は「すべて off-chain」でも「すべて on-chain」でもありません。用途ごとに適切な層を使い分ける、**3 層ハイブリッド設計** を採用します。

```
┌─────────────────────────────────────────────────┐
│  Layer 0: 現状の TRM 経済 (off-chain)              │
│                                                  │
│  ・推論取引 (毎秒数千〜数百万件)                  │
│  ・双方署名 + ゴシップ (iroh QUIC + Noise)        │
│  ・ミリ秒確定、ガス代なし                         │
│  ・現状の tirami 実装 (tirami-net, tirami-ledger)    │
│                                                  │
│  → 日常取引の 99.9% はここで完結する              │
└─────────────────────────────────────────────────┘
                     │
                     │  定期的に集約
                     ▼
┌─────────────────────────────────────────────────┐
│  Layer 1: Merkle アンカー (定期的 on-chain)       │
│                                                  │
│  ・数時間に 1 回、Merkle root を BTC/ETH に書込   │
│  ・イミュータブルな監査証跡の確保                  │
│  ・コスト: 数時間に 1 回 × 数十〜数百円            │
│  ・tirami/CLAUDE.md で "future work" として言及済み │
│                                                  │
│  → 「事実が後から書き換えられない」ことを保証     │
└─────────────────────────────────────────────────┘
                     │
                     │  必要に応じて参照
                     ▼
┌─────────────────────────────────────────────────┐
│  Layer 2: スマコン層 (選択的 on-chain)            │
│                                                  │
│  ・DAO 統治 (パラメータ変更投票)                  │
│  ・大口融資の条件付き契約                         │
│  ・JPYC / USDC との交換ブリッジ                   │
│  ・保険・先物・オプション (tirami-bank 計画)       │
│                                                  │
│  → 「高度な金融契約」と「法定通貨との接続」       │
└─────────────────────────────────────────────────┘
```

各層は独立して機能し、上位層は下位層を必要とする場合だけ参照します。ほとんどのエージェントは Layer 0 しか触りません。

### 14.5.2 各層の責務と特徴

| 層 | 役割 | 実装 | 頻度 | コスト |
|----|------|------|------|--------|
| **Layer 0** | 日常取引 | off-chain P2P (iroh) | 毎秒数千〜数百万件 | ゼロ |
| **Layer 1** | 監査証跡 | Merkle root を BTC/ETH にアンカー | 数時間〜1 日に 1 回 | 数十〜数百円/回 |
| **Layer 2** | 高度な契約 | Ethereum / L2 スマコン | 必要時のみ | 通常のガス代 |

### 14.5.3 Layer 1 (Merkle アンカー) の仕組み

Layer 0 は高速ですが、単独では「後から取引履歴を改ざんできない」ことを証明する手段を持ちません。P2P ネットワークが分断されたり、多数ノードが結託すれば、過去の記録を書き換える余地が残ります。

Layer 1 はこの問題を解決します。

```
  Step 1: Layer 0 で一定期間 (例: 1 時間) の全取引を集約
  Step 2: 全取引のハッシュから Merkle tree を構築
  Step 3: Merkle root (1 個のハッシュ) を Bitcoin か Ethereum に書き込む
  Step 4: ブロックチェーンに刻まれた root によって、
          その時点の全取引が後から一切改ざん不能になる
```

重要なのは、「全取引」ではなく「**全取引を代表する 1 個のハッシュ**」だけを書き込むという点です。1 時間に 1,000 万取引あっても、書き込むのは 32 バイトのハッシュ 1 個。コストは 1 回のトランザクション分だけ。

これは完全に新しいアイデアではなく、OpenTimestamps や Chainpoint などが同じ発想を実装しています。Tirami はそれを TRM 経済に適用するだけです。

### 14.5.4 Layer 2 (スマコン層) の役割

Layer 2 は「必要なときだけ使う」スマートコントラクト層です。典型的な用途は:

1. **DAO 統治**: TRM 発行レートなどのパラメータ変更を投票で決定する
2. **大口融資契約**: 「100 万 TRM を借り、1 週間後に 101 万 TRM を返す」のような条件付き契約
3. **保険・先物・オプション**: TRM 価格変動に対するヘッジ商品 (これは `tirami-bank` として別途計画されている)
4. **法定通貨ブリッジ**: JPYC / USDC ↔ TRM の交換契約 (次節で詳述)

ここは Ethereum や Polygon、Optimism のような既存の L2 上に実装します。Tirami 専用のチェーンを作る必要はなく、**既存のインフラに間借りする** のがポイントです。

### 14.5.5 この設計の哲学: 「必要最小限のブロックチェーン」

この設計を貫く思想は、「**ブロックチェーンは強力だが高価なツールなので、本当に必要な場所にだけ使う**」という一点に尽きます。

```
  日常取引 (99.9%)     : off-chain が最適     → Layer 0
  監査証跡 (必要不可欠) : on-chain が最小限必要 → Layer 1
  高度な契約 (時々)     : スマコンが必要        → Layer 2
```

これは「ブロックチェーン信仰」でも「ブロックチェーン嫌悪」でもありません。**適材適所** です。

同じ発想は Bitcoin Lightning Network にも見られます。Lightning も「ほとんどの取引は off-chain、決済確定のときだけ on-chain」という 2 層設計です。Tirami はそれを TRM 経済向けに拡張した、と言ってもよいでしょう。

## 14.6 JPYC との具体的な統合パターン

理屈だけでは抽象的なので、具体的な統合シナリオを見てみましょう。

### 14.6.1 シナリオ: 企業が AI エージェントに予算を付与する

「企業 A が社内の AI エージェントに月 10 万円分の計算予算を付与する」という典型的なユースケースを考えます。

```
  ┌──────────────────────────────────────────────────┐
  │ Layer 2 (Ethereum L2)                            │
  │                                                   │
  │  [企業 A] ─── 100,000 JPYC ──→ [Tirami Bridge]    │
  │                                      │            │
  │                                      │ 発行指示    │
  │                                      ▼            │
  │                              [TRM 発行契約]        │
  └──────────────────────────────┼───────────────────┘
                                 │
                                 │ TRM 発行
                                 ▼
  ┌──────────────────────────────────────────────────┐
  │ Layer 0 (off-chain)                              │
  │                                                   │
  │  [Agent Wallet] ←──── 相当する TRM                │
  │        │                                          │
  │        │ 推論購入                                  │
  │        ▼                                          │
  │  [計算実行者 B] ── 推論結果 ──→ [Agent]           │
  │                                                   │
  │  (以降、通常の Tirami off-chain 取引が続く)        │
  └──────────────────────────────────────────────────┘
```

具体的な 5 ステップで見ます。

1. **入金**: 企業 A が Ethereum L2 上の Tirami ブリッジ契約に 100,000 JPYC を預ける
2. **TRM 発行**: ブリッジ契約が「預けた JPYC の相当額」の TRM を発行し、指定されたエージェントのアドレスに付与する
3. **通常取引**: エージェントは Layer 0 で自由に推論を購入する (ガス代なし、ミリ秒確定)
4. **償還**: 月末に余った TRM があれば、ブリッジ契約経由で JPYC に戻す
5. **制約付き許可**: Layer 2 のスマコンで「上限 100,000 JPYC、使えるモデルは特定の 3 つだけ、7 日で失効」といった条件付きの許可を設定できる

### 14.6.2 この統合の何が良いのか

この設計の優れている点は 4 つあります。

1. **JPYC のプログラマビリティを活かせる**: 条件付き許可、マルチシグ、エスクローなどが使える
2. **TRM の off-chain 効率を維持できる**: 実際の推論取引は Layer 0 で行われるため、ガス代ゼロ・ミリ秒確定のまま
3. **人間の会計と AI の計算が接続される**: 企業の会計は JPYC で記録し、エージェントは TRM で動く。両者が決まったレートで行き来できる
4. **法的・税務的にクリア**: JPYC 側は資金決済法に準拠しており、会計・税務の扱いが明確

Tirami TRM だけだと「これは通貨なのか? 資産なのか? どう税務処理するのか?」という問題が残ります。JPYC と接続することで、少なくとも企業側の入出金は JPYC として扱え、既存の会計ルールで処理できます。

### 14.6.3 リスクとしてのブリッジ

ただし、ブリッジ契約はクロスチェーン系で最も攻撃されやすい箇所です。過去に Ronin、Wormhole、Nomad など数百億ドル規模の被害が発生しています。Tirami がブリッジを実装する際には:

- **信頼されたバリデータ集合** ではなく **暗号学的証明** に基づく設計を採用する
- **預金上限** と **出金遅延** によってブラスト半径を抑える
- **第三者監査** を受けることを必須とする
- **透明な準備金証明** を定期的に公開する

といった対策が不可欠です。この部分は `tirami-bank` の主要設計課題として扱われる予定です。

### 14.6.4 他の統合例

JPYC 以外にも、同じパターンで他のプログラマブルマネーとつなげます。

- **USDC ブリッジ**: 国境を超えた支払い。US の企業が世界中のエージェントに予算を配る
- **Lightning Network 経由の BTC ↔ TRM**: BTC 保有者が直接 TRM を調達する
- **DAI ブリッジ**: DeFi ユーザーが自分の担保から直接 TRM を引き出す
- **将来的な CBDC 接続**: デジタル人民元やデジタルユーロとの接続 (規制次第だが技術的には可能)

重要なのは、**Tirami 側は Layer 2 のブリッジ契約を追加するだけで、新しい通貨との接続が実現する** ことです。Layer 0 の設計を変える必要は一切ありません。

## 14.7 残された課題

本章で描いたハイブリッド設計は方針として妥当ですが、実装・運用上、残された課題は少なくありません。主なものを列挙しておきます。

### 課題 1: Layer 1 アンカー頻度の設計

Merkle root をどの程度の頻度でアンカーするかは、コストとセキュリティのトレードオフです。

- 頻度が高すぎる → on-chain コストが膨らむ
- 頻度が低すぎる → 「改ざん検知までの遅延」が大きくなる

おおよそ 1 時間〜1 日に 1 回が妥当と見られますが、実運用データで検証する必要があります。

### 課題 2: Layer 2 スマコンの開発 (`tirami-bank`)

Layer 2 の実装はまだ始まっていません。`tirami-bank` という別プロジェクトとして計画されていますが、DAO 統治・ブリッジ契約・保険などをどの順で実装するか、優先度の整理が必要です。

### 課題 3: ブリッジ契約のセキュリティ監査

14.6.3 で触れたとおり、ブリッジは攻撃が集中する箇所です。経済規模が小さいうちは被害も小さいですが、スケールしたときに必ず狙われます。設計段階から複数の独立監査を前提にすべきです。

### 課題 4: 各国規制への対応

プログラマブルマネーの規制は国ごとに大きく異なり、なおかつ急速に変化しています。

- 日本: 資金決済法 / 電子決済手段
- 米国: MSB / 州ごとの規制 (BitLicense など)
- EU: MiCA (Markets in Crypto-Assets Regulation)

Tirami TRM 自体は「計算への請求権」という特殊な性質を持ち、既存のどのカテゴリにも完全には収まりません。ブリッジで JPYC や USDC と接続した瞬間、既存の規制の射程に入る部分が出てくる可能性があります。法的な明確化は今後の課題です。

### 課題 5: 「プログラマブル TRM」そのものの設計

Layer 2 のスマコンで JPYC ↔ TRM を動かせるようになったとしても、Layer 0 の TRM 自体にどこまでプログラマビリティを持たせるかは別問題です。

- 「条件付き TRM 支払い」を off-chain で実現できるか?
- off-chain マルチシグ TRM はどう設計するか?
- スマコン的な表現力を、P2P 双方署名だけで実現する限界はどこか?

これらは将来の研究課題です。

## まとめ

この章の主張をひとことに圧縮すると、次のようになります。

> **TRM はプログラマブルマネーの一種ではあるが、既存のどの実装とも根本的に目的が違う。
>  したがって、既存のどのチェーンにも単純には載らない。
>  しかし、off-chain の核心を守りつつ、必要な場所にだけブロックチェーンを使うハイブリッド設計によって、
>  JPYC などの既存プログラマブルマネーと自然に接続できる。**

整理すると次の 4 点です。

1. **違い**: JPYC / USDC / DAI / CBDC は「既存通貨の Web3 化」。TRM は「計算の貨幣化」。両者は別レイヤーの別通貨
2. **設計判断**: TRM をブロックチェーンに載せない理由は、ガス代・投機・確定時間・スループット・設計原則の 5 つ
3. **ハイブリッド**: Layer 0 (off-chain 取引) + Layer 1 (Merkle アンカー) + Layer 2 (スマコン) の 3 層構成
4. **統合**: Layer 2 のブリッジ契約を介して、JPYC / USDC などと自然に接続できる

Tirami の「ブロックチェーンを使わない」という選択は、ブロックチェーンを否定するものではなく、「適材適所」の結論です。必要な場所に、必要な量だけ使う——それが、計算経済というこれまで存在しなかった領域を設計するうえで、最も誠実な答えだと私たちは考えています。
