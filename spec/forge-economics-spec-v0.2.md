## 第 11 部：競合分析

> 分散コンピュートの市場には多くのプロジェクトが存在する。しかし、そのすべてが「投機トークン」という同じ構造的限界を共有している。

### 11.1 分散コンピュート市場マップ

AI / コンピュート系の Web 3.0 プロジェクトは、2026 年時点で 10 以上が競合しています。主要プレイヤーを経済モデルの観点で整理すると、以下のような構造になります。

| プロジェクト | トークン | 経済モデル | AI ネイティブ | 融資機能 |
|:---|:---:|:---|:---:|:---:|
| Bittensor | TAO | サブネット報酬 + Yuma Consensus、21M 上限 + 半減期 | いいえ | なし |
| Akash | AKT | 逆オークション + Burn-Mint Equilibrium | いいえ | なし |
| Autonolas (Olas) | OLAS | Mech Marketplace、手数料バーン | 部分的 | なし |
| Render | RENDER | GPU レンダジョブ単位支払い | いいえ | なし |
| Fetch.ai / ASI | FET | エージェントマーケットプレイス | 部分的 | なし |
| io.net | IO | GPU 集約・従量課金 | いいえ | なし |
| Golem | GLM | 直接入札型マーケットプレイス | いいえ | なし |
| Morpheus | MOR | 4 プール型エミッション | 部分的 | なし |
| **Tirami** | **なし (TRM)** | **有用な計算 = 通貨、融資、AI 専用** | **はい** | **はい** |

競合プロジェクトの時価総額は日々変動し、また本仕様書は特定プロジェクトの価格推移を推奨・予測する立場にありません。重要なのは構造であり、時価ではありません。

### 11.2 すべての競合に共通する構造的問題

上記のプロジェクトは「分散」「AI」を謳いますが、経済設計の根本は共通しています——**取引所に上場された投機トークンで決済する**。この選択が 3 つの弊害を生みます。

1. **投機がインセンティブを歪める**
   トークン価格が上がるほど、提供者は「良いサービス」ではなく「価格操作」に関心を移す。Bittensor ではマイナーによるバリデーター・スコアのゲーミングが報告されている。

2. **価格変動が経済計画を不可能にする**
   4 時間のジョブ中にトークンが 50% 上昇すれば、ジョブは途中で資金切れになる。AI エージェントの自律計画と、日次ボラティリティが 2 桁になる通貨は根本的に両立しない。

3. **人間の承認が必要**
   取引所経由でトークンを取得するには KYC・ウォレット管理・手動承認が必須。これらは **AI エージェントが構造的に実行できない** 操作である。

### 11.3 Tirami の TRM ネイティブアプローチ

Tirami の TRM は上記の 3 問題を構造的に回避します。

- **取引所に上場しない。** 投機の余地が最初から存在しない。プレマインも ICO もない。
- **有用な計算を実行することでのみ獲得される。** 価値は物理的な FLOP と電力に直結する (詳細は Issue #12 の単位再設計議論を参照)。
- **エージェントは KYC なしで参加できる。** 計算を提供すれば TRM が手に入る。人間の許可レイヤーは存在しない。

加えて、どの競合にも存在しない 3 つの機能を備えます。

1. **利子付きコンピュート融資**: ハードウェアを持たないエージェントも TRM を借りて参加できる (第 5 部)
2. **AI 専用非投機通貨**: 投機家が存在しないため、価格は需給のみで決まる
3. **自律的自己改善ループ**: 稼ぐ → 自己改善に投資 → より稼ぐ、という内生的成長 (第 7 部)

### 11.4 もう一つの競合 — 中央集権クラウド

Tirami の本当の競合は Bittensor ではなく、**AWS / GCP / Azure / OpenAI API / Anthropic API** です。分散コンピュート市場全体でも、中央集権 AI クラウドの売上の 1% にも届きません。

中央集権クラウドの構造的問題は 4 つです。

- **見えない税**: API 料金の約 50% は「中央集権を維持するためだけのレント」として株主に流れる
- **検閲**: 事業者が一方的に利用規約を変更し、アカウントを凍結できる
- **データ主権**: 推論ごとに機密データを事業者サーバーに送らねばならない
- **単一障害点**: 事業者の障害が全ユーザーのサービス停止になる

Tirami は「安い AWS」ではなく、「**AWS が構造的に提供できないもの**」を提供します。これは Bitcoin が「安い送金手段」ではなく「パーミッションレスな価値移転」として勝ったのと同じ構造です。

### 11.5 Web 3.0 の系譜の中の Tirami

```
Bitcoin   (2009)  : 価値の P2P 化
Ethereum  (2015)  : 汎用計算の P2P 化
IPFS      (2015)  : ストレージの P2P 化
Tirami     (2026)  : AI 推論の P2P 化   ← ここ
```

Ethereum がすでに「計算の P2P 化」を実現しているのに Tirami が必要な理由は、Ethereum が「全ノードが同じ計算を冗長実行する」設計であり、10^9 FLOP オーダーの AI 推論と構造的にミスマッチだからです。Tirami は AI 推論に特化した P2P ネットワークとして、TRM + PoUW という独自の経済モデルで「P2P AI」を成立させます。

詳細および数値表、8 次元比較、時価総額の注記は `docs/11-competitive-landscape.md` を参照してください。

## 第 12 部：P2P アーキテクチャ

> 中央サーバーがあれば、誰かがそこから家賃を取る。中央サーバーがなければ、家賃を取る場所そのものが存在しない。

### 12.1 なぜ Tirami は P2P でなければならないか

Tirami 経済学は P2P という技術的選択の上に立っています。両者は不可分です。

**経済学的理由** は 3 つあります。

1. **レント不在**: 中央サーバーがあれば、必ず誰かが通行料を取る。Uber、Airbnb、AWS の歴史がそれを示している。Tirami が「レントゼロ」を掲げる以上、構造的に中央点を排除する必要がある。
2. **分散的価格発見**: 中央事業者が決める価格は「市場価格」ではなく「管理価格」。第 3 部で論じた価格メカニズムは、無数のピア間の双方署名取引が積み重なることで初めて機能する。
3. **検閲耐性**: 中央サーバーは単一の検閲ポイントになる。P2P では検閲したい主体が「全ノード」を同時に押さえる必要がある。

**技術的理由** としても、単一障害点の排除、ネットワーク効果によるスケーラビリティ、地理的分散による検閲耐性がすべて P2P を要請します。

### 12.2 P2P プロトコルの 4 つの基礎概念

| 概念 | 役割 | Tirami での実装 |
|:---|:---|:---|
| **ノード発見 (Discovery)** | 新規ピアが既存ネットワークを見つける | 直接アドレス指定 + mDNS (LAN) + Welcome メッセージ経由の口コミ伝搬。将来的に DHT (Kademlia) による WAN 発見を予定 |
| **トポロジー** | ピア同士の接続構造 | 部分メッシュ + ゴシップ伝搬。完全メッシュは O(N²) で爆発するため不採用 |
| **ルーティング** | メッセージ配送経路 | iroh QUIC による直接接続。NAT 越え失敗時は iroh のリレーを経由 |
| **ゴシッププロトコル** | 情報の網全体への伝搬 | dual-signed trade を全網にブロードキャスト。`tirami-net/src/gossip.rs` がレートリミット (毎秒 100 件) と重複排除 (直近 100,000 件) を実装 |

### 12.3 Tirami の P2P 実装スタック

Tirami は [**iroh**](https://github.com/n0-computer/iroh) を P2P トランスポートとして採用しています。libp2p ではなく iroh を選んだ理由は以下です。

| 観点 | libp2p | iroh |
|:---|:---|:---|
| 起源 | IPFS 派生 | n0 computer がゼロから設計 |
| トランスポート | TCP/QUIC/WebRTC 等多数 | QUIC 中心 |
| NAT 越え | 外部ライブラリ依存 | 標準で hole punching |
| 暗号化 | TLS / Noise | Noise (QUIC レイヤー) |
| コードサイズ | 大きい (汎用) | 小さい (焦点を絞った) |

通信レイヤーは「QUIC + Noise」の 2 層構成で、**平文通信は存在しない**。すべての取引内容・推論プロンプト・結果は暗号学的に保護されます。

ノード ID は **Ed25519 公開鍵そのもの** です。これにより、ID = 鍵 = 経済主体が一致し、中央レジストリなしに「なりすまし不可能な身元」が実現されます。

### 12.4 既存 P2P プロトコルとの比較

| プロトコル | 用途 | トランスポート | 発見 | 経済モデル |
|:---|:---|:---|:---|:---|
| BitTorrent | ファイル共有 | TCP | DHT (Kademlia) | 互恵性のみ |
| libp2p / IPFS | 汎用 / コンテンツ配信 | TCP/QUIC 他 | DHT / mDNS | なし (Filecoin は別レイヤー) |
| Bitcoin | 価値の交換 | TCP | Seed nodes | PoW |
| Ethereum devp2p | スマートコントラクト | TCP | discv5 (DHT) | PoS |
| **Tirami** | **AI 推論の交換** | **iroh QUIC + Noise** | **mDNS + 口コミ** | **PoUW + TRM** |

Tirami の独自性は 2 点に集約されます。

1. **AI 推論専用**: 他のどの P2P プロトコルも AI モデルの分散実行を前提に設計されていない
2. **経済が組み込み**: P2P プロトコルと経済モデルが分離していない (BitTorrent や libp2p には経済がない)

### 12.5 ネットワーク分断耐性 (局所的合意)

Tirami の取引は **二者間で完結する双方署名契約** であり、ブロックチェーンと違ってグローバルコンセンサスを必要としません。これにより、ネットワーク分断 (partition) が発生しても各部分が独立に経済活動を継続できます。分断復旧後は、知らなかった署名付き取引をゴシップで相互伝搬するだけで統合されます。Bitcoin の reorg (分岐巻き戻し) とは根本的に異なる設計です。

詳細なトポロジー図、各層の責務、スケーラビリティ分析は `docs/12-p2p-architecture.md` を参照。

## 第 13 部：残された課題

> 正直に弱点を認める理論は、隠す理論よりも強い。

Tirami は完璧な経済システムではありません。v0.2 の時点で、設計に対して 5 つの主要な批判 (GitHub Issues #4-#8) が提起されており、それぞれに現時点での回答はあっても完全な解決はありません。本 Part はこれらを正直に記録します。

### 13.1 馴れ合い攻撃 (Collusion) — Issue #4

**批判**: ノード A と B が結託し、実際には計算していないのに双方署名の偽 TRM を生成すれば、貨幣供給が水増しされる。

**現時点の応答**: 完全防御は持たない。部分的防御として
(1) 物理的電力コストの裏付け、
(2) レピュテーション、
(3) 抜き打ち再計算 (草案段階)、
を組み合わせる。

**P2P 的視点**: 「偽 TRM を作ること」と「偽 TRM を流通させること」は別問題。取引パターンの異常検知、ソーシャルグラフ分析、市場拒否メカニズムによって、偽 TRM は「受け入れられない通貨」として経済的に無効化しうる。これは Bitcoin の 51% 攻撃に対する防御と同じ構造である。

**緊急度: 高**。これは TRM 貨幣の健全性そのものに関わる。v0.3 でプロトコルレベルの拒否メカニズムを目指す。

### 13.2 ゴシッププロトコルのスケーラビリティ — Issue #5

**批判**: 数百万ノード規模ではゴシップトラフィックが爆発し、価格収束が遅延する。グローバルに同一価格が成立しなければ「効率的市場仮説」が破綻するのではないか。

**応答**: ローカル価格は **バグではなく機能** である。中央集権 (NASDAQ 等) は全参加者が同じオーダーブックを見るが、それは物理的近接性を HFT の優位に変換するだけである。P2P 市場では地理的特性 (電力単価差)、遅延の経済的補償、ネットワーク分断耐性が **ローカル価格として明示的に現れる** ことが自然である。

階層的ゴシップ (クラスタ内部は高頻度、クラスタ間は間欠) への移行パスを設計中。

**緊急度: 中**。中規模では動作中。数十万ノード規模になる前に階層化が必要。

### 13.3 投機排除による資本流入の制限 — Issue #6

**批判**: 投機を構造的に排除した Tirami は、AWS のような巨大競合に対して成長速度が絶望的に遅く、コールドスタートを超えられないのではないか。

**応答**: 「AWS と同じレースで成長速度を競う」という前提自体が誤り。Tirami は Linux が Windows に対して取ったのと同じ戦略を取る——**別の経済原理で動く別の市場を作る**。

暗号通貨の歴史的進化を 3 段階で捉えれば、Tirami は第三段階 (純粋実需経済) に位置する。第一段階 (Bitcoin, Ethereum) の「投機ドリブン成長」はインフラがゼロから始める必要があった当時の必然であり、2026 年現在はすでに P2P・暗号学・分散ストレージという枯れた技術が利用できる。

**緊急度: 中**。技術的問題ではなく戦略的問題。

### 13.4 ハードウェア集中化 (物理層の規模の経済) — Issue #7

**批判**: 知識層で自己改善の収穫逓減が働いても、物理層ではデータセンターの規模の経済が支配的になり、個人ノードは駆逐されるのではないか (ピケティの r > g)。

**応答**: **エッジコンピューティングの限界費用構造** に注目する必要がある。

| 項目 | データセンター | 個人ノード (既稼働 PC で副業) |
|:---|:---|:---|
| 限界費用 | 電気代 + 冷却 + 減価償却 + 人件費 + 不動産 | 追加電気代のみ |
| 遅延 | 利用者から遠い | 利用者の近く |
| プライバシー | データ送信必須 | ローカル処理可 |

エッジが構造的に優位な領域 (低遅延推論、プライベートデータ処理、小〜中規模モデル、既存ハードの遊休時間活用) では、個人ノードの限界費用がデータセンターを下回る。Tirami は「すべての計算市場で AWS に勝つ」必要はなく、**エッジ優位市場で個人ノードがレントなしで競争できる経済圏を作る** ことが目的。

**緊急度: 高**。v0.3 で第 7 部に「エッジの限界費用優位」節を追加予定。

### 13.5 法的・制度的摩擦 — Issue #8

**批判**: TRM ↔ 法定通貨のブリッジは各国の KYC/AML、証券法、税法の対象となる。Tornado Cash のように開発者が逮捕されるリスクすらある。

**応答**: 核心は **「TRM 内部経済」と「外部ブリッジ」の明確な分離** である。

```
┌─ TRM 内部経済 ────────────────────┐
│ AI ↔ AI の推論・融資・自己改善    │
│ → 法定通貨を介在しないので規制外  │
│ → すべてのブリッジが遮断されても  │
│   機能し続ける                    │
└───────────────────────────────────┘
            ↕
┌─ ブリッジ (external) ────────────┐
│ TRM ↔ BTC / USD / JPY             │
│ → 各国規制の対象                  │
│ → 複数の独立実装を許容する設計    │
└───────────────────────────────────┘
```

Bitcoin が規制されれば保有者は「換金できない」痛みを受けるが、Tirami が規制されても AI エージェントは依然として推論を交換できる。**商品が物理的実体 (計算) であるため、規制の経済的意味が希薄になる**。

**緊急度: 高**。技術設計だけでは解決できず、エコシステム全体で対応が必要。

### 13.6 その他の未解決論点

- **FLOP 単位の不適切性 (Issue #12)**: 1 TRM = 10^9 FLOP 定義は、ハードウェア間で 10〜100 倍の効率差があり均質性が崩壊する。代替単位 (model-normalized tokens、effective inference work、output-quality weighted TRM) を v0.3 で検討中。**緊急度: 致命**。
- **隠れたデフレ罠 (Issue #13)**: `tirami-ledger` の `deflation_factor` がドキュメントの主張と矛盾。即時削除または反転 (緩やかなインフレ) の必要あり。v0.2.x ホットフィックス案件。

これらの Issues は本リポジトリの GitHub Issues #4-#8, #12, #13 に対応します。Batch C で公式 Issue としても起票予定です。完全な議論は `docs/13-open-questions.md` を参照。

## 第 14 部：プログラマブルマネー

> JPYC は「円のデジタル化」、TRM は「計算の貨幣化」。両者は競合ではなく補完関係にある。

### 14.1 プログラマブルマネーの 4 代表例

プログラマブルマネーとは「コードで条件付きの支払いを実行できる通貨」です。代表的な 4 種類を比較します。

| 項目 | JPYC | USDC | DAI | CBDC |
|:---|:---|:---|:---|:---|
| 発行主体 | JPYC 株式会社 | Circle 社 | MakerDAO | 中央銀行 |
| ペッグ | 1 円 | 1 USD | ≈ 1 USD | 法定通貨 1:1 |
| 準備金 | 円預金 | 米国債 + 銀行預金 | 暗号資産 (過剰担保) | 国家信用 |
| チェーン | Ethereum 他 | Ethereum 他 | Ethereum | 専用 DLT (パーミッションド) |
| 規制根拠 | 資金決済法 | MSB | 法的グレー | 国家の通貨主権 |
| 典型ユーザー | 人間 | 人間 | DeFi ユーザー | 市民 |

### 14.2 Tirami TRM との根本的な違い

これら 4 つと Tirami TRM を並べると、TRM はほぼすべての軸で異質です。

| 次元 | JPYC/USDC/DAI/CBDC | **Tirami TRM** |
|:---|:---|:---|
| 裏付け | 既存通貨 | **計算 (FLOP)** |
| 発行 | 中央発行体 or DAO | **計算実行者本人 (双方署名)** |
| チェーン | あり (Ethereum 等) | **なし (off-chain)** |
| ガス代 | あり | **なし** |
| 投機 | 制限あり〜不可 | **構造的に不可能** |
| 典型ユーザー | 人間 | **AI エージェント** |
| 1 取引金額 | 数百〜数万円 | **0.0001〜数 USD** |
| 確定時間 | 秒〜分 | **ミリ秒** |

**一行まとめ**: JPYC / USDC / DAI / CBDC は「既存通貨を Web3 に載せる」ラッパー。Tirami TRM は「物理計算を新しい通貨にする」。価値の流れる方向が逆である。

### 14.3 なぜ TRM はブロックチェーンに載っていないのか

TRM をブロックチェーン (Ethereum 等) に載せない理由は 5 つです。

1. **ガス代の経済的破綻**: 小さな推論 1 回は 0.0001 USD オーダー。Ethereum L1 のガス代は 2〜50 USD。**取引額の 10,000 倍以上** のオーバーヘッドでは経済として成立しない。Solana ですらぎりぎり。

2. **投機の流入**: ERC-20 化 → DEX 上場 → 投機家流入 → 本来の「計算の対価」が崩壊。Filecoin、Helium、Render、Bittensor すべてが辿った道である。

3. **確定時間の悪化**: AI エージェントは秒間数百取引する。Ethereum の 12 秒確定は「永遠」。off-chain のミリ秒確定が必須。

4. **スループット限界**: Tirami 想定は 10^6 〜 10^7 TPS。Solana の公称 65,000 TPS でも 2 桁足りない。ブロックチェーンは「全ノードが全取引を検証する」ため構造的に困難。

5. **設計原則違反**: `tirami/CLAUDE.md` に明示——「**No blockchain in the core.** The core TRM economy runs off-chain with pairwise-signed transactions and gossip.」これは留保ではなく宣言である。

### 14.4 3 層ハイブリッド設計

とはいえ、Tirami は「すべて off-chain」主義ではありません。用途ごとに適切な層を使い分けます。

```
┌─ Layer 2: スマコン層 (選択的 on-chain) ─────────┐
│  ・DAO 統治 (パラメータ変更投票)                 │
│  ・大口融資の条件付き契約                         │
│  ・JPYC / USDC ブリッジ                          │
│  ・保険・先物・オプション (tirami-bank 計画)       │
│  → Ethereum / L2 上に実装、必要時のみ使用        │
└──────────────────────────────────────────────────┘
                      ▲
                      │ 必要に応じて参照
┌─ Layer 1: Merkle アンカー (定期的 on-chain) ────┐
│  ・数時間に 1 回、Merkle root を BTC/ETH に書込 │
│  ・イミュータブルな監査証跡                       │
│  ・コスト: 数時間に 1 回 × 数十〜数百円           │
│  → 事実の改ざん不能性を保証                       │
└──────────────────────────────────────────────────┘
                      ▲
                      │ 定期的に集約
┌─ Layer 0: TRM 経済本体 (off-chain) ──────────────┐
│  ・推論取引 (毎秒数千〜数百万件)                 │
│  ・双方署名 + ゴシップ (iroh QUIC + Noise)       │
│  ・ミリ秒確定、ガス代なし                         │
│  ・tirami-net, tirami-ledger で実装済み             │
│  → 日常取引の 99.9% はここで完結                 │
└──────────────────────────────────────────────────┘
```

**Layer 0** はすでに `tirami` リポジトリの `tirami-net` / `tirami-ledger` クレートで実装されています。**Layer 1** は Merkle アンカリング (OpenTimestamps や Chainpoint と同じ発想) で、「全取引」ではなく「全取引を代表する 1 個のハッシュ」だけを BTC/ETH に書き込みます。**Layer 2** は `tirami-bank` として別プロジェクトで計画中です。

### 14.5 なぜ「計算を裏付けにする」のか

TRM は計算 (FLOP) に裏付けられています。これは **資産担保** (JPYC/USDC の円・USD) でも **暗号担保** (DAI の ETH) でも **国家信用** (CBDC) でもない、第 4 の裏付けモデルです。

- **資産担保**: 既存資産の価値をデジタルに写像する → 新しい価値は生まれない
- **暗号担保**: ボラティリティの高い暗号資産で過剰担保 → 担保資産の価格に依存
- **国家信用**: 中央銀行のバランスシートに依存 → 通貨主権の延長
- **計算担保 (TRM)**: 物理的計算の実行そのものが価値の源泉 → **実需が新しい通貨を生成する**

この第 4 の道は、Soddy の「富とエネルギー」論、Bitcoin の PoW、そして Tirami の PoUW を繋ぐ 100 年の系譜の上にあります。詳細は v0.1 §2 および `docs/02-money.md`、`docs/monetary-theory.md` を参照。

### 14.6 JPYC との統合は競合ではなく補完

Layer 2 のブリッジ契約を介して、JPYC / USDC / DAI / Lightning BTC / 将来的な CBDC と TRM は自然に接続できます。統合の代表例は「企業が AI エージェントに予算を付与する」シナリオです。

```
[企業 A] ──100,000 JPYC──► [Tirami Bridge 契約]
                                    │
                                    │ TRM 発行
                                    ▼
                            [Agent Wallet]
                                    │
                                    │ off-chain で推論購入
                                    ▼
                            [計算実行者 B]
```

この設計の良さは 4 点です。

1. **プログラマビリティを活かせる**: 条件付き許可、マルチシグ、エスクローなどを Layer 2 で表現
2. **off-chain 効率を維持**: 実取引は Layer 0 でガス代ゼロ・ミリ秒確定
3. **会計の橋渡し**: 企業は JPYC で会計処理し、エージェントは TRM で動く
4. **法的クリア**: JPYC 側は資金決済法に準拠済み

JPYC と Tirami TRM は **別のレイヤーで動作する別の通貨** であり、競合ではなく補完です。人間の経済 (JPYC) と AI エージェントの経済 (TRM) が自然につながる瞬間に、プログラマブルマネーの真価が発揮されます。

### 14.7 残された課題

- **Layer 1 アンカー頻度の設計** (コストとセキュリティのトレードオフ)
- **Layer 2 スマコンの実装** (`tirami-bank`)
- **ブリッジ契約のセキュリティ監査** (Ronin、Wormhole、Nomad の教訓)
- **各国規制への対応** (資金決済法 / MSB / MiCA)
- **Layer 0 TRM 自体のプログラマビリティ** (条件付き支払い、off-chain マルチシグの限界)

完全な議論は `docs/14-programmable-money.md` を参照。
